# 重构验证清单

## 1. 无障碍恢复 / Shizuku

- Shizuku 已授权时，手动让无障碍服务失联后重新打开 App，应自动恢复
- 快捷方式触发识别时，无障碍失联应先尝试恢复，再执行识别
- 截图磁贴触发识别时，无障碍失联应先尝试恢复，再执行识别
- 悬浮窗磁贴触发时，无障碍失联应先尝试恢复，再执行逻辑
- Root / Shizuku 模式下，设置页或主页不应出现“设置已开启但服务实例为空仍判定可用”的假活状态

## 2. 胶囊 / 模板渲染

- 列车事件胶囊标题应优先显示真实内容，如 `A8 检票`，不应偶发显示模板原文 `检票口 检票`
- 航班、取件、取餐、取票、寄件等模板型事件也不应出现模板原文直出
- 后台停留较久后恢复、切前台后刷新、规则刷新后刷新，胶囊标题仍应稳定

## 3. 提醒 / 通知

- 冷启动场景下，提醒广播触发时不应因事件列表尚未加载而被误判为事件不存在
- Room 主存启用后，已有事件的提醒应正常重建
- `开始时` 提醒不应重复触发两次
- 胶囊关闭时，未显式选择 `开始时` 的事件仍应保留默认开始时通知
- 删除事件、归档事件、还原事件后，对应提醒应正确取消或重建

## 4. 日历同步

- 反向同步频率应恢复为 60 秒级，不应再出现 1 秒高频轮询
- 应用升级后（覆盖安装），提醒与同步应能恢复
- 开启同步后，手动同步与自动同步都应正常
- 系统日历新增/修改/删除事件后，App 侧应正常新增、更新、删除
- 已归档事件不应因系统日历回流而“僵尸复活"

## 5. 归档 / 恢复

- 单个事件归档后，应从活跃列表移除并进入归档列表
- 归档事件还原后，应回到活跃列表并移出归档列表
- 删除单个归档事件时，系统日历映射应同步清理
- 清空归档时，系统日历映射应批量清理
- 重复实例、循环事件相关删除/归档逻辑不应误删 Room 记录

## 6. 课程 / 导入导出

- 课程新增、编辑、删除、覆盖导入、追加导入都应正常
- WakeUp 课表导入应正常，学期开始日期应自动标准化
- 应用备份格式课程导入后，课程表和作息配置应正确恢复
- 日程导出应包含活跃事件和归档事件
- 日程导入去重应正常，归档状态应正确保留或更新

## 7. 回归关注点

- `AppRepository` 已做较多职责拆分，重点回归所有“保存后立即刷新”的页面
- 重点观察：主页、归档页、设置页、悬浮窗、胶囊、提醒广播、系统日历双向同步
- 已知限制：未开启自启动或电池无限制时，部分系统在滑卡后可能拦截提醒，这属于系统后台策略，不一定是 App 故障

# 2026-03-26 开发记录 (v33版本)

## 一、本次实现范围

1. **Room 主存全面默认化**：新版本默认直接以 Room/SQLite 作为日程主存，旧 JSON 不再作为正式数据源
2. **升级后主动迁移全量日程**：应用启动时立即迁移活动日程、重复日程、归档日程到数据库
3. **旧内部 JSON 统一退役**：迁移完成后清理 `events.json`、`archives.json` 及旧备份，保留 `*.room.bak` 兜底
4. **实验室开关收口**：移除 Room 主存/读取模式与 CalendarSync V2 开关，避免用户进入半迁移状态
5. **升级防重同步**：在重新接管系统日历前，先补建映射，尽量避免升级后把已有日程再创建一份

## 三、关键逻辑摘要

### 3.1 存储策略切换
- `isRoomMainEnabled()` 直接返回开启态，后续新增/编辑日程都以 Room 为主
- JSON 不再承担主存职责，仅保留 `events.room.bak`、`archives.room.bak` 作为应急兜底

### 3.2 启动即迁移
- `refreshData()` 中除了活动日程/重复日程迁移，还会主动读取 `archives.json` 并执行 `LegacyArchiveMigrator`
- 不再依赖用户进入归档页触发迁移，避免“活动已进库、归档仍留在 JSON” 的半完成状态

### 3.3 旧 JSON 退役策略
- 当 events / recurring / archives 三类迁移标记都完成后，立即删除内部旧文件：
  - `events.json`
  - `events.json.bak`
  - `archives.json`
  - `archives.json.bak`
- 用户手动导出的 JSON 不在 `filesDir` 管理范围内，不会被清理

### 3.4 防止升级后日程复制
- V2 正向同步前先执行两层“补建映射”保护：
  1. 先把旧 `sync_data.json` 中已有 mapping 补写进 `calendar_sync_map`
  2. 再扫描系统日历内已有事件，优先按 `appId` 匹配，兜底仅对 App 托管事件按指纹/重复规则基线哈希匹配
- 若首次自动同步时发现还没有映射，会先做一次反向吸收，再进入正向写入，降低“升级后整批重复创建”的风险

## 四、预期用户升级行为

1. 安装新版本并启动 App
2. 后台静默把旧 JSON 日程迁移到 Room
3. 同步模块优先补齐旧映射/识别系统日历中已存在的托管事件
4. 确认数据库已接管后清理旧内部 JSON
5. 后续新增、编辑、归档都继续走数据库主存

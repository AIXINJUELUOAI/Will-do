## 二、多模态 AI 问题排查与结论

### 现象

- 用户在使用多模态 AI 分析图片日程时，始终提示“返回格式错误”

### 排查过程

1. 先比对 v43 重构记录与相关文件历史，确认昨日改动主线并未直接修改多模态识别主流程
2. 检查 `ApiModelProvider` 与 `RecognitionProcessor` 当前实现，发现客户端对多种返回结构兼容不足
3. 增加多模态请求 / 响应日志后，抓到真实响应：中转站返回 `chat.completion`，但 `message.content = null`

### 结论

- 当前用户报错的直接根因不是本地 JSON 解析崩溃，而是**中转站 + 当前模型组合返回空内容**
- 旧版本大概率也会失败，只是过去提示不够明确，更像“静默失败”
- 因此本轮主要目标从“修解析崩溃”转为“增强兼容、补强日志、把失败原因讲清楚”

## 三、AI 响应兼容与日志增强

### 1. 响应解析兼容增强

调整 `ApiModelProvider.parseModelResponse()`：

- 兼容 `message.content` 为字符串
- 兼容 `message.content` 为数组分片
- 兼容部分接口返回 `choices[0].text`
- 补充对 `responses API` 风格 `output[].content[]` 的文本提取兜底

### 2. JSON 清洗与事件解析增强

调整 `RecognitionProcessor`：

- `cleanJsonString()` 支持从代码块、前后说明文字中提取 JSON 主体
- 事件解析兼容：
  - 顶层 `{ "events": [...] }`
  - 顶层单对象
  - 顶层数组

### 3. 关键日志新增

新增日志点用于快速判断问题归因：

- 多模态请求摘要（模型、URL、图片大小、prompt 长度）
- 请求封装摘要（reasoning、payload 大小、data URL 前缀）
- 原始服务响应
- 空内容时的 `finish_reason / native_finish_reason / reasoning_content / refusal / message`
- `RecognitionProcessor` 层的最终失败详情

### 当前收益

- 下次再收到用户反馈时，可以直接通过日志区分：
  - 中转站空响应
  - 模型不支持图片
  - 参数不兼容
  - 响应结构异常
  - 本地解析失败

## 四、AI 失败提示体系重构

### 目标

将原本散落在 `RecognitionProcessor` 中的错误提示逻辑抽离，形成统一入口，符合 v43 的“门面化 / 统一收口”方向。

### 实现

新增文件：

- `core/ai/AiFailureMapper.kt`

统一收口以下错误类型：

- `CONFIG`
- `NETWORK`
- `HTTP`
- `PARSE`
- `UNKNOWN`

### 文案策略调整

为适配当前 Toast / Snackbar 展示宽度，所有 AI 错误文案统一压缩为短句：

- `AI未配置`
- `模型无输出`
- `响应缺choices`
- `响应缺分片`
- `响应缺candidates`
- `响应为空`
- `响应格式错`
- `网络超时`
- `地址无效`
- `网络不可达`
- `网络异常`
- `参数错误(400)` / `接口不存在(404)` / `网关异常(502)` 等

### 效果

- 用户侧提示更短，不再经常被 UI 截断
- 开发侧可通过状态码与关键词快速定位问题方向
- 后续文本 AI / OCR / 多模态都可复用同一套错误映射逻辑

## 五、主页 Toast 被底栏遮挡修复

### 现象

- 首页底部 Toast 会被集成底栏遮挡，或与底栏贴得过近

### 根因

- `HomeScreen` 计算 Snackbar 避让高度时，仅使用了底栏声明高度
- `IntegratedFloatingBar` 实际绘制中还额外包含：
  - 组件高度补偿
  - 底部阴影安全区
- 两侧使用的“底栏高度口径”不一致

### 修复

在 `IntegratedFloatingBar.kt` 中抽出统一常量：

- `IntegratedFloatingBarExtraHeight`
- `IntegratedFloatingBarShadowPadding`
- `IntegratedFloatingBarVisualHeight`
- `IntegratedFloatingBarToastGap`

并让 `HomeScreen` 的 `SnackbarHost` 避让高度使用同一套视觉口径。

### 效果

- Toast 不再被底栏遮挡
- 与底栏之间保留稳定间距
- 后续若继续调底栏高度或阴影，只需改共享常量

## 六、模型配置页输入框交互优化

### 问题 1：输入框有内容时很难点击

#### 根因

- 原实现使用裸 `BasicTextField`
- 外层点击与内部文本命中区域不一致
- 文本右对齐后，实际有效交互体验更差

#### 过程

1. 尝试用 `TextField` 内核替换，点击问题明显改善
2. 但 `TextField` 默认高度与内部留白偏大，和原卡片风格不符
3. 最终改为“紧凑版 `BasicTextField` + 统一点击容器 + 明确交互源”方案

### 问题 2：点击后光标总落在开头

#### 根因

- 外层点击层会抢走原始点击事件
- 文本控件拿不到真实落点，无法按点击位置计算光标

#### 修复

- 输入内部状态从 `String` 切为 `TextFieldValue`
- 去掉会抢点击定位的外层点击逻辑
- 将光标与选区控制权交还给 `BasicTextField`

### 效果

- 输入框在保留紧凑卡片风格的前提下，点击更稳定
- API Key / 模型名称 / API 地址在有内容时更容易聚焦
- 光标落点更符合用户点击位置，不再总回到开头

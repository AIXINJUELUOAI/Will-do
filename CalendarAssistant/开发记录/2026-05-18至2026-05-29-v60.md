### 3.2 位置来源与手动位置选择

位置策略经过多轮讨论后定为：

- 保留 `自动定位`。
- 保留 `手动定位`。
- 移除 `自动定位优先`，避免用户理解成本过高。
- 手动位置采用一个 Sheet 内三级联动选择：省/自治区/直辖市/港澳台 -> 市/州 -> 区县。
- 仅覆盖中国大陆、港澳台，不做海外城市。
- 手动位置选择后保存和风 `LocationID`、经纬度、行政区名称等信息。

### 3.3 天气详情页

天气详情页承载更高信息密度：

- 当前天气卡片展示温度、天气文本、体感温度、湿度、风力、能见度等。
- 未来 24 小时改为横向温度折线图。
- 折线图在关键点显示温度和天气 icon。
- 未来一周改成紧凑列表。
- 每日详情默认收起，点击后展开降水、风力、紫外线、日出日落等信息。
- 预警默认显示摘要，点击展开后显示详情、防御指南、生效/过期时间。

### 3.4 天气预警通知与实况通知

天气通知架构经过重构：

- 普通天气通知统一走 `NotificationCenter`。
- 天气实况通知统一走 `CapsuleCenter -> CapsuleStateManager`。
- 官方预警和风险提醒均可进入实况胶囊。
- 实况胶囊受 `偏好设置 -> 实况胶囊通知` 总开关控制。
- 天气实况持续时长由 30 分钟缩短为 1 分钟，定位为提醒而非长期驻留状态。
- 官方预警按类型去重，例如同类 `黄色雷电预警` 只保留最贴近定位区域或最新的一条。
- 天气实况自然复用现有设备路由：HyperOS 超级岛、Flyme 实况通知、原生实况通知。

这轮还明确了：

- `官方天气预警` 是气象部门正式发布的预警信号。
- `天气风险提醒` 是根据未来天气趋势推断的非官方风险提示。

## 四、桌面小组件

### 4.1 架构设计

小组件开发前明确要求贴合项目架构，避免孤立实现。

最终采用：

- `WidgetCenter`：统一刷新、分发尺寸、读取设置。
- `WidgetScheduleQueryApi` / `LocalWidgetScheduleQueryApi`：生成小组件展示快照。
- `WidgetScheduleSnapshot`：统一保存小组件可展示日程表。
- `ScheduleWidgetProvider`：只做 Android 小组件入口。
- `ScheduleWidgetRenderer`：负责 `RemoteViews` 渲染。

核心文件包括：

- `app/src/main/java/com/antgskds/calendarassistant/core/center/WidgetCenter.kt`
- `app/src/main/java/com/antgskds/calendarassistant/core/query/WidgetScheduleQueryApi.kt`
- `app/src/main/java/com/antgskds/calendarassistant/data/model/WidgetScheduleSnapshot.kt`
- `app/src/main/java/com/antgskds/calendarassistant/widget/ScheduleWidgetProvider.kt`
- `app/src/main/java/com/antgskds/calendarassistant/widget/ScheduleWidgetRenderer.kt`

### 4.2 多尺寸布局

小组件采用一个 Provider 多个尺寸布局：

- `2x1`：显示今日未来第一条未完成日程；今天无未完成日程时显示今日空闲。
- `2x2`：显示今日未来两条未完成日程；今天无未完成日程时显示今日空闲。
- `4x2`：左侧显示今天第一条，右侧最多显示两条；今天没有时左侧显示今日空闲，右侧继续显示未来日程。
- `4x4`：按统一可展示日程表顺序显示今日未完成和未来日程。

经过测试后，数据规则统一为：

- 先生成统一的 `visibleEntries`。
- 过滤归档、便签、已完成、已过期事件。
- 展开重复日程实例。
- 所有尺寸都从同一张表切片展示，避免不同尺寸显示规则不一致。

### 4.3 样式与设置页

小组件视觉参考 OPPO 日历组件，并做了多轮适配：

- 修复 RemoteViews 不支持普通 `<View>` 导致的小组件载入失败。
- Android 12+ 改用 `RemoteViews(Map<SizeF, RemoteViews>)`，提升尺寸选择稳定性。
- 修复外层 R 角割裂问题，避免应用圆角和 launcher 圆角叠加。
- 字号改为根据应用 `uiSize` 与系统字体缩放动态计算。
- 4x2 右侧优先显示今天后续日程，贴近 OPPO 逻辑。

设置层面新增：

- 桌面小组件设置页。
- 2x1 / 2x2 / 4x2 / 4x4 实时预览。
- 小组件主题：跟随软件、浅色、深色。
- 背景透明度调节。
- 预览区使用斜向黑白条纹背景，便于观察透明度。

## 五、附件系统与数据备份

### 5.1 从 description 标记迁移到附件表

此前识别来源截图依赖 `description` 中的 `[img:...]` 标记，容易被系统日历同步回流覆盖。

本轮改为独立附件体系：

- 新增 `event_attachments` 表。
- 新增 `EventAttachment` 模型。
- 新增 `EventAttachmentManager`。
- 截图来源作为 `source = recognition_image` 的附件保存。
- 用户手动上传附件作为 `source = manual` 保存。
- `description` 回归纯备注文本，不再承担附件元数据。

核心文件包括：

- `app/src/main/java/com/antgskds/calendarassistant/calendar/models/EventAttachment.kt`
- `app/src/main/java/com/antgskds/calendarassistant/core/attachment/EventAttachmentManager.kt`
- `app/src/main/java/com/antgskds/calendarassistant/ui/dialogs/AddEventDialog.kt`

需要特别说明：

- 如果历史截图路径已被系统日历同步覆盖，本轮迁移无法凭空恢复原截图。
- 本轮只能保证后续新识别/新添加附件不再依赖 `description`，降低再次丢失概率。

### 5.2 附件绑定策略

附件绑定经历了几轮修正：

1. 仅使用临时 id 的方案不稳定。
2. 讨论过 `localUid / draftSessionId` 方案，但改动较大。
3. 最终采用 `RoomID 主绑定 + title/time event_key 兜底`。

当前策略：

- 已有日程：附件直接绑定 Room `Event.id`。
- 新建日程未确认前：附件先绑定 `event_key(title + startTS + endTS + timezone)`。
- 创建成功后：拿到 RoomID，再把附件转绑到 `event_id`。
- 打开日程时：优先按 `event_id` 查附件；查不到时再用 `event_key` 兜底，并自动修复回当前 RoomID。
- 删除日程时主动删除附件和软件目录中的副本，不依赖数据库外键级联。

### 5.3 Dialog 附件体验

新建/编辑日程 Dialog 增加附件区：

- 可添加文件。
- 可打开附件。
- 可删除附件。
- 图片附件支持直接预览。
- 新建日程时临时附件可在保存后绑定到新日程。
- 预览图高度调整为更适合截图查看，减少裁切。

修复过的问题：

- 主线程访问 Room 导致打开备份页崩溃。
- 点击日程加载附件时主线程访问 Room 崩溃。
- 删除日程时附件删除逻辑在主线程执行导致崩溃。
- 新建日程关闭 Dialog 后附件丢失。
- Room schema 改动未升版本导致 identity hash 不一致。

### 5.4 数据备份升级

数据备份页从“日程数据”升级为“数据备份”：

- 课程数据导入导出保持原逻辑。
- 日程数据改为统一备份入口。
- 导出时弹 Sheet 勾选：日程、设置、附件、提示词。
- 不含附件时导出 JSON。
- 包含附件时导出 ZIP。
- 导入时自动识别 JSON / ZIP，并让用户选择恢复哪些部分。
- 设置备份包含 API Key、模型地址、天气 Key 等完整配置，不做脱敏。

核心文件包括：

- `app/src/main/java/com/antgskds/calendarassistant/core/center/BackupCenter.kt`

## 六、剪贴板码类识别与 InstantCode 统一解析

### 6.1 需求背景

新增剪贴板取件码 / 取餐码 / 取票码 / 寄件码识别。

最终交互定为：

- 无 Shizuku / Root：用户打开软件时读取一次剪贴板，识别到码类后弹 `FloatingActionCard` 确认是否入库。
- 有 Shizuku / Root：后台日志监听剪贴板变化，识别到码类后直接入库。
- 悬浮窗不再读取剪贴板，避免和悬浮窗输入框、确认卡片打架。

### 6.2 正则与解析抽取

为了避免短信和剪贴板各维护一套码类正则，本轮抽出统一模块：

- `InstantCodeType.kt`
- `InstantCodePatterns.kt`
- `InstantCodeParser.kt`
- `InstantCodeCandidate.kt`

旧的短信分析逻辑改成兼容转发层，短信和剪贴板共用核心正则与解析逻辑。

支持类型包括：

- 取件：`EventTags.PICKUP`
- 取餐：`EventTags.FOOD`
- 取票：`EventTags.TICKET`
- 寄件：`EventTags.SENDER`

解析结果统一转为 `RecognitionDraft`，继续走现有入库管线。

### 6.3 Center 层接入

新增 `ClipboardCodeCenter`：

- 负责读取剪贴板。
- 负责去重和忽略记录。
- 负责判断当前是否有 Shizuku / Root 权限。
- 无权限时暴露 pending prompt 给 UI 弹确认卡。
- 有权限时直接通过 `IngestCommandApi` 入库。

实验室新增：

- `剪贴板码类识别（实验）` 开关。
- 文案明确说明：有 Shizuku / Root 可后台自动识别，无权限时仅打开软件后确认识别。

### 10.3 日志导出语义调整

## 十二、新建/编辑日程类型化输入

本轮承接 `2026-05-01-v57.md`，完成了新建/编辑日程 Dialog 的类型化输入第一版。

### 12.1 结构化字段

在 `AddEventDialog` 中新增内部类型配置：

- 日程：保留地点、备注。
- 列车：车次、检票口、座位号。
- 航班：航班号、登机口、座位号。
- 打车：颜色、车型、车牌。
- 取件：取件码、品牌、位置。
- 取餐：取餐码、品牌、位置。
- 取票：取票码、品牌、位置。
- 寄件：寄件码、品牌、地点。

选择特殊类型后：

- 底部 `地点 / 备注` 区域切换为该类型的结构化字段。
- 底部主按钮从 `确定` 切换为 `完成`。
- 点击 `完成` 时拼接为现有 `description` 微格式。
- 完成后回到普通 `地点 / 备注` 输入区。
- 再点 `确定` 才真正保存日程。

继续复用：

- `RecognitionRuleCatalog.formatDescription(...)`
- `RuleMatchingEngine.resolvePayload(...)`
- `RuleMatchingEngine.splitFields(...)`

不改数据库，不新增实体。

### 12.2 类型 Chip 分页交互

类型行经过多轮调整，最终定为分页式横滑：

- 第 1 页：`日程/重复`、`列车`、`航班`、`打车`
- 第 2 页：`取件`、`取餐`、`取票`、`寄件`

规则：

- 点击当前类型 Chip 展开。
- 每页严格 4 个等宽 Chip，不露出第 5 个。
- 左右滑动按页切换。
- 当前选中 Chip 主色填充。
- 未选中 Chip 为透明背景 + 线框 + 文本。
- 第一格根据重复规则动态显示 `日程` 或 `重复`，但本质仍对应 `EventTags.GENERAL`。
- 选择第一格只切回普通日程类型，不改变重复规则。

## 十四、已知风险与后续待办

1. 历史来源截图如果已被系统日历同步覆盖，无法自动恢复，只能保证后续附件链路稳定。
2. 小组件在不同 Launcher 上尺寸映射和圆角裁剪表现仍需真机持续验证。
3. 天气预警、风险提醒、实况通知在 HyperOS / Flyme / 原生 Android 上仍需设备侧持续回归。
4. 预测性返回已限定在页面级和自定义底部弹窗，侧边栏、悬浮窗、系统 AlertDialog 暂不做。
5. 剪贴板后台监听依赖 Shizuku / Root 和 logcat 行为，不保证所有 ROM 都稳定。
6. 标准版已移除本地模型入口，但 `ai-engine` 相关工程代码仍在，需要后续按版本策略决定是否继续保留。
7. 诊断日志只有在 `AppLogger` 覆盖到的路径上才有内容，后续应逐步补充更多关键业务链路日志。
8. 日程类型化 Dialog 目前仍使用 `tag + description 微格式`，如果后续类型字段继续膨胀，可能需要考虑独立结构化事实源。

## 十五、阶段结论

v60 阶段是一次功能密集扩展和交互补强：

- 天气、小组件、附件、备份、日志让应用从“识别日程工具”继续扩展为更完整的个人日程系统。
- 剪贴板码类识别和统一 `InstantCode` 解析降低了短信、剪贴板、本地 prompt 路由的重复维护成本。
- 预测性返回、触感反馈、弹窗动效、日程类型化输入提升了日常操作体验。
- 数据备份和诊断日志提升了迁移、排障和测试效率。

下一阶段建议优先围绕真机回归、稳定性修复和文档同步展开，避免继续在未充分验证的基础上叠加大功能。

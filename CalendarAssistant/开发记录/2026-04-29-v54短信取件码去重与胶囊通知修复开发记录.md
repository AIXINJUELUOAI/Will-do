# 2026-04-29 开发记录 (v54)

## 一、本次目标

本次围绕用户反馈的三个问题进行排查和修复：

1. 短信自动提取取件码时，偶发创建两个相同日程。
2. Flyme 状态栏取件码小胶囊颜色饱和度过高，且与 App 内标签色不一致。

## 二、短信取件码重复入库修复

### 1) 问题确认

短信取件码原先有三个入口会直接尝试入库：

- `SmsReceiver`：短信广播入口。
- `SmsContentObserver`：短信数据库监听入口。
- `SmsNotificationListenerService`：通知监听兜底入口。

三个入口都可能调用 `ingestSmsPickup()`。旧去重依赖 `scheduleCenter.events.value`，该 StateFlow 在新增日程后通过异步刷新更新，因此多个入口近乎同时触发时，后一个入口可能看不到刚入库的第一条，导致重复创建。

此外旧去重依赖完整 `description` 一致。不同入口拿到的短信正文或通知正文可能略有差异，同一取件码也可能绕过去重。

### 2) 设计方案

保留三个入口作为信号源，但不让它们直接竞争入库。入口只提交候选短信，由统一协调器串行处理。

入口优先级与延迟：

```text
SMS_RECEIVER           0ms
CONTENT_OBSERVER       600ms
NOTIFICATION_LISTENER  1500ms
```

处理规则：

- 前一个入口解析失败、入库异常、入库 API 未就绪时，后一个入口继续尝试。
- 前一个入口创建成功或被最终去重判定为已存在时，后续同取件码入口跳过。
- 同一取件码用稳定指纹终止后续尝试，而不是依赖完整短信正文。

### 3) 代码落点

新增：

- `app/src/main/java/com/antgskds/calendarassistant/core/sms/SmsPickupIngestCoordinator.kt`
  - 使用 `Channel` 串行处理入口候选。
  - 维护短期 `terminalFingerprints`，防止同一取件码被后续入口重复入库。

- `app/src/main/java/com/antgskds/calendarassistant/core/sms/SmsPickupFingerprint.kt`
  - 基于 `tag + normalized code` 生成稳定指纹。
  - 支持取件、取餐、取票、寄件等码类事件。
  - 优先从 `RuleMatchingEngine.resolvePayload()` 解析 payload。

调整：

- `App.kt`
  - 增加 `smsPickupIngestCoordinator` 单例。
  - `SmsContentObserver` 改为获取协调器。

- `SmsReceiver.kt`
  - 不再直接解析和入库，改为提交 `SmsPickupSource.SMS_RECEIVER`。

- `SmsContentObserver.kt`
  - 不再直接解析和入库，改为提交 `SmsPickupSource.CONTENT_OBSERVER`。

- `SmsNotificationListenerService.kt`
  - 不再直接解析和入库，改为提交 `SmsPickupSource.NOTIFICATION_LISTENER`。

- `ImportCenter.kt`
  - 为 `ingestSmsPickup()` 增加 `Mutex`，保证“查重 + 新增”原子串行。
  - 去重改为读取最新数据库快照，不再依赖可能滞后的 `scheduleCenter.events.value`。
  - 同时支持稳定指纹去重与旧 description 兜底去重。

- `ScheduleCenter.kt`
  - 增加 `getLatestActiveEvents()`，供入库前读取最新活跃事件。


## 三、Flyme 取件码小胶囊颜色修复

### 1) 问题确认

Flyme 状态栏小胶囊不是系统随机分配颜色。代码中取件码胶囊曾单独写死 `Color.GREEN`，即高饱和纯绿，而普通日程使用事件自身颜色。

旧链路：

```text
CapsuleStateManager
  -> pickup color = Color.GREEN
  -> FlymeCapsuleProvider.builder.setColor(item.color)
  -> notification.live.capsuleBgColor = color
```

因此只有取件码颜色与 App 内标签色不一致，并且在 Flyme 小胶囊上看不清。

### 2) 修复内容

- `CapsuleStateManager.kt`
  - 单个取件码胶囊改为使用 `event.color`。
  - 聚合取件码胶囊使用第一个有效取件事件颜色。
  - 无有效颜色时使用柔和兜底色 `rgb(180, 195, 161)`。

- `FlymeCapsuleProvider.kt`
  - 对胶囊背景色做归一化，避免无效颜色落到不可读状态。
  - 根据背景相对亮度计算黑/白内容色。
  - 小胶囊图标也使用同一内容色 tint。
  - `notification.live.capsuleContentColor` 不再固定白色。

### 3) 验证情况

Flyme 颜色修复阶段已执行：

```text
./gradlew.bat :app:compileDebugKotlin
```

结果：编译通过，仅有 Android deprecated API 警告。

## 四、删除取件码后冒普通通知问题排查与临时修复

### 1) 问题确认

用户反馈：开启胶囊通知后，测试短信取件码，删除该取件码时，胶囊消失后会冒出一个普通通知。

排查后判断，该问题很可能是 v53 重复日程通知修复后暴露出的旧链路问题：

- v53 为了修复重复实例普通提醒，在 `AndroidManifest.xml` 注册了 `EventReminderReceiver`。
- 单次事件的 Store 提醒链路原本也会向 `EventReminderReceiver` 发送普通提醒广播。
- `ReminderPolicy.effectiveReminders()` 会强制补 `0 分钟提醒`。
- 短信取件码通常 `startTS` 约等于当前时间，创建时会触发即时普通提醒广播。
- 以前 `EventReminderReceiver` 未注册时，这条广播可能无效果；注册后普通通知开始真正显示。

不应取消 `EventReminderReceiver` 注册，否则会破坏 v53 的重复日程通知修复。

### 2) 修复策略

保留 v53 设计，增加普通通知链路防线：

- 胶囊开启时，单次事件不再通过 `EventReminderReceiver` 调度或发送普通提醒。
- `EventReminderReceiver` 收到广播后重新查询最新数据库，确认事件仍可展示。
- 事件删除、归档、完成、过期时，同时清理普通通知残留。
- `AlarmReceiver` 不再只依赖 `scheduleCenter.events.value` 校验单次事件，改为读取最新数据库事件。

### 3) 代码落点

- `ReminderStoreNode.kt`
  - `rebuildForEvent()` 中：胶囊开启时直接取消该事件普通提醒与普通通知残留，并返回。
  - 新增 `cancelForInactiveEvent()`，统一取消闹钟、普通通知残留和 active marker。
  - 新增 `cancelDisplayedNotificationsForEvent()`，取消可能存在的普通通知 ID，包括旧 `EventReminderReceiver` ID 和取件码初始通知偏移 ID。

- `EventReminderReceiver.kt`
  - 改为 `goAsync()` 后在 IO 线程处理。
  - 展示前读取 `calendarCenter.getEvent(eventId)`。
  - 胶囊开启、事件不存在、已归档、非待办、已过期时直接跳过并取消通知。

- `StoreRootNode.kt`
  - 删除、归档、自动归档、恢复重复实例等取消提醒场景改用 `cancelForInactiveEvent()`。

- `AlarmReceiver.kt`
  - 单次事件校验改为读取最新数据库事件。
  - 重复实例校验仍保留 v53 的 `rec:${parentId}:${occurrenceTs}` 逻辑，但事件列表来源改为数据库活跃事件。

### 4) 当前验证状态

通知链路修复后尝试执行：

```text

```

该命令在用户要求“先这样吧”后中止，因此本阶段尚未完成最终编译验证。

## 五、需要继续验证的场景

后续继续时建议优先验证以下场景，避免修复一边破坏另一边：

1. 胶囊开启 + 短信取件码：只出现胶囊，不出现普通通知。
2. 胶囊开启 + 删除取件码：胶囊消失，不再冒普通通知。
3. 胶囊关闭 + 短信取件码：普通通知仍可正常出现。
4. 胶囊关闭 + 重复日程实例：到点普通通知仍正常。
5. 胶囊开启 + 重复日程实例：到点进入胶囊链路。
6. 删除重复实例或母事件：旧闹钟不会继续发普通通知或胶囊刷新。

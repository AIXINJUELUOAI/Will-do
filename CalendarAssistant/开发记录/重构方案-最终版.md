# Will do v2.0 重构方案（最终版）

> 目标：对齐系统日历重复模型、规则引擎化业务逻辑、保留双 Prompt 稳定性并支持扩展。

## 1. 目标与原则

- 对齐系统日历 Events/Instances 模型，彻底解决重复日程同步乱象。
- 规则引擎化渲染与交互，消除 if(tag==xxx) 硬编码。
- 保留双 Prompt（Schedule + Pickup）稳定性，同时支持规则补丁扩展。

## 2. 数据模型（Room）

### 2.1 核心表

**event_masters**
- masterId (PK)
- ruleId? (null=普通日程)
- title, description, location, color
- rrule?
- syncId?
- eventType, remindersJson
- isImportant, sourceImagePath?, skipCalendarSync
- createdAt, updatedAt
- source (如 "legacy_json")

**event_instances**
- instanceId (PK)
- masterId (FK, onDelete=CASCADE)
- startTime, endTime (epochMillis)
- currentStateId
- completedAt?, archivedAt?
- syncFingerprint (UNIQUE)
- isSynced, isCancelled

**event_excluded_dates**
- excludedId (PK)
- masterId (FK, onDelete=CASCADE)
- excludedStartTime (epochMillis)
- 索引：(masterId, excludedStartTime)

**event_rules**
- ruleId, name, isEnabled, appliesToSchedule
- aiTag, aiPrompt
- extractionConfigJson
- iconSourceJson
- initialStateId

**event_states**
- stateId, ruleId, name, isTerminal, displayTemplate

**event_transitions**
- transitionId, ruleId
- fromStateId, toStateId
- actionLabel, autoTransitionAfterMillis?

**calendar_sync_map**
- localMasterId (PK)
- systemEventId
- calendarId (系统日历 _id)
- accountName, accountType
- displayName (仅用于 UI)
- lastSyncHash

### 2.2 约束与索引

- event_instances(masterId) 索引
- event_instances(startTime, endTime) 复合索引
- event_instances(syncFingerprint) UNIQUE
- calendar_sync_map(calendarId, systemEventId) 复合索引
- 时间统一存 Long (epochMillis)
- LocalDate/Color/列表/Json 通过 TypeConverters

## 3. 规则引擎（RuleMatchingEngine）

- 输入：EventInstance + EventMaster + EventRule
- 协议头：description 必须以 **【ruleId】** 开头
- 提取方式：delimiter / regex / ai
- 字段缺失：使用默认占位符（如 "未获取"）
- 渲染：按 currentStateId 选 displayTemplate
- 动作：按 event_transitions 生成按钮
- 终态：EventState.isTerminal=true，记录 completedAt 并消失
- 兜底：未识别或已删除 ruleId -> DefaultRule（普通日程）

## 4. AI 识别流（双 Prompt + 规则补丁）

- Schedule Prompt = 固定 Header + 固定 Schedule Prompt + 动态规则补丁
- Pickup Prompt 固定（高频场景稳定优先）
- Multimodal Prompt 同样支持规则补丁
- 规则补丁来源：isEnabled && appliesToSchedule

## 5. 同步 V2（对齐系统日历）

### 5.1 App -> System
- master 有 rrule -> 写 Events
- 单次事件 -> 写单条
- 保存 syncId

### 5.2 System -> App
- 通过 CalendarContract.Instances 查询
- syncFingerprint = hash(masterId + startTime + endTime)
- lastSyncHash 计算范围：title + startTime + endTime + location + description + rrule（trim 后）
- 缺失实例：写入 event_excluded_dates 或标记 isCancelled

### 5.3 防抖
- ContentObserver 触发后改用 WorkManager
- OneTimeWork + 5s delay + UniqueWork(REPLACE)

## 6. 视图查询优化（Room）

- FullInstance POJO
  - @Embedded instance
  - @Relation master
  - @Relation rule?
- @Transaction 查询一次拿齐渲染所需数据
- 高并发列表可考虑 DatabaseView 预连接

## 7. 重复日程展示与归档策略

### 7.1 展示原则
- UI 只展示实例（event_instances），不直接展示 master
- isCancelled 默认不展示（归档页可筛选）
- 重复日程加“重复”标识

### 7.2 自动归档
- 条件：endTime < now - 24h -> archivedAt
- 仅归档实例，不归档 master
- 不影响后续实例生成

### 7.3 归档页显示
- archivedAt != null 的实例可见
- 已过期样式：灰色 + 划线，仅标注“已过期”
- 不标记“未完成”

## 8. 迁移策略（JSON -> Room）

- App 启动时主动迁移活动日程、重复日程、归档日程
- 迁移：每条 JSON -> master + instance，重复规则拆分为 master/instance/excludedDates
- master.source = "legacy_json"
- SharedPreferences 记录 events/recurring/archives 三类迁移完成时间
- 迁移完成后删除内部旧文件：`events.json`、`events.json.bak`、`archives.json`、`archives.json.bak`
- 保留 `events.room.bak`、`archives.room.bak` 作为 Room 主存兜底备份

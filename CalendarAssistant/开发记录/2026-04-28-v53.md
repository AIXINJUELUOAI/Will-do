# 2026-04-28 开发记录 (v53)

## 一、本次目标

修复用户反馈的重复日程实例到点提醒问题，并让重复实例与普通日程保持一致的通知策略：

1. 胶囊通知关闭时，重复实例发普通通知。
2. 胶囊通知开启时，重复实例进入胶囊通知链路。
3. 重复实例使用虚拟 id 注册闹钟，避免同一母事件下不同日期实例互相覆盖。
4. 保持重复实例的完成、签到动作可用。
5. 不执行编译，修复完成后等待用户验证。

---

## 二、问题确认

### 1) 初始问题

重复日程的实例到点后不会发出通知。排查后确认：

- `ReminderStoreNode` 为重复实例注册的是 `EventReminderReceiver`。
- `EventReminderReceiver` 未在 `AndroidManifest.xml` 中注册。
- 因此 AlarmManager 到点后无法投递到目标广播接收器。

### 2) 用户复测后的新问题

补上接收器后，重复实例只能发普通通知，不能像普通日程一样进入胶囊链路。

根因是重复实例和普通日程使用了两套不同的提醒路径：

```text
普通日程
  -> NotificationScheduler
  -> AlarmReceiver
  -> ReminderCenter / CapsuleCenter
  -> 普通通知或胶囊通知

重复实例
  -> ReminderStoreNode
  -> EventReminderReceiver
  -> 普通通知
```

此外，`CapsuleStateManager` 原本会直接跳过 `event.isRecurring` 的重复母事件，也不会展开虚拟重复实例，所以即使闹钟触发了胶囊刷新，胶囊状态里也找不到对应实例。

## 三、设计决策

### 1) 重复实例虚拟 id

重复实例统一使用展示层已有的稳定 key：

```text
rec:${parentId}:${occurrenceTs}
```

该 id 用于：

- AlarmManager 注册 key。
- PendingIntent requestCode 计算。
- 胶囊通知 id / action id。
- 完成、签到动作回传。

### 2) requestCode 维度

requestCode 不只使用实例 id，还加入 action 与 marker，避免同一实例的不同闹钟互相覆盖：

```text
${instanceKey}:${action}:${marker}:${type}
```

其中 action 包括：

- `ACTION_REMINDER`
- `ACTION_CAPSULE_START`
- `ACTION_REFRESH_CAPSULE`
- `ACTION_CAPSULE_END`

### 3) 通知分流规则

重复实例按当前设置动态选择注册内容：

- 胶囊关闭：注册普通提醒闹钟，触发 `AlarmReceiver.ACTION_REMINDER`。
- 胶囊开启且全局提前提醒关闭：在实例开始时间注册 `ACTION_CAPSULE_START`，在结束时间注册 `ACTION_CAPSULE_END`。
- 胶囊开启且全局提前提醒开启：提前注册 `ACTION_CAPSULE_START`，准点注册 `ACTION_REFRESH_CAPSULE`，结束注册 `ACTION_CAPSULE_END`。

## 四、代码落点

### 1) Manifest 注册

文件：`app/src/main/AndroidManifest.xml`

- 注册 `.calendar.receivers.EventReminderReceiver`。
- 保留旧普通日程 `AlarmReceiver` 注册。

### 2) 重复实例提醒注册

文件：`app/src/main/java/com/antgskds/calendarassistant/store/reminder/ReminderStoreNode.kt`

主要调整：

- 重复实例不再固定走 `EventReminderReceiver`。
- 根据 `settings.isLiveCapsuleEnabled` 生成普通通知或胶囊闹钟规格。
- Intent 使用实例虚拟 id 作为 `EVENT_ID`，同时携带 `EVENT_PARENT_ID` 与 `EVENT_OCCURRENCE_TS`。
- 旧 requestCode 会在刷新时取消，避免设置切换后残留普通通知或胶囊闹钟。
- 对 `setAndAllowWhileIdle` 增加异常保护，缺少 exact alarm 能力时回退到 inexact alarm。

### 3) 胶囊状态支持重复实例

文件：`app/src/main/java/com/antgskds/calendarassistant/core/capsule/CapsuleStateManager.kt`

主要调整：

- 使用 `ScheduleDisplayHelper.buildDisplayItems()` 展开近期日程实例。
- 对 `RecurringOccurrence` 构造临时 `Event`，让现有 `CapsuleMessageComposer` 和规则展示逻辑复用。
- 胶囊 item 使用虚拟 id 与独立 notification id。
- 保持普通单次事件、已物化子事件、重复虚拟实例统一进入活跃胶囊筛选。

### 4) AlarmReceiver 实例校验

文件：`app/src/main/java/com/antgskds/calendarassistant/service/receiver/AlarmReceiver.kt`

主要调整：

- `EVENT_ID` 以 `rec:` 开头时按重复实例处理。
- 根据 `EVENT_PARENT_ID`、`EVENT_OCCURRENCE_TS` 和展示展开结果确认实例仍然有效。
- 父事件不存在、实例被排除或已不在展开结果中时，跳过通知/胶囊刷新。
- 胶囊关闭但收到 capsule start 时，兜底发普通通知并保留实例时间、地点、颜色等信息。

### 5) 通知动作支持虚拟实例

文件：`app/src/main/java/com/antgskds/calendarassistant/service/receiver/EventActionReceiver.kt`

主要调整：

- 支持解析 `rec:${parentId}:${occurrenceTs}`。
- 将动作路由到 `ActionTarget.RecurringOccurrence`。
- 胶囊上的完成、签到按钮可以作用于重复实例。

## 五、行为覆盖

### 1) 胶囊关闭

重复实例到点时走：

```text
ReminderStoreNode
  -> AlarmReceiver.ACTION_REMINDER
  -> NotificationCenter.showStandardNotification
```

预期：发出普通高优先级通知。

### 2) 胶囊开启，无全局提前提醒

重复实例到点时走：

```text
ReminderStoreNode
  -> AlarmReceiver.ACTION_CAPSULE_START
  -> ReminderCenter.refreshCapsuleState
  -> CapsuleStateManager 展开重复实例
  -> 胶囊通知
```

预期：实例开始时出现胶囊，实例结束后移除。

### 3) 胶囊开启，有全局提前提醒

重复实例会注册三类闹钟：

- 提前时间：启动胶囊。
- 日程准点：刷新胶囊文案。
- 结束时间：刷新并移除过期胶囊。

预期：与普通日程的胶囊时序一致。

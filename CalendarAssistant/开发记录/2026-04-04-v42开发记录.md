## 二、Bug 修复详情

### Bug 1：打车规则待办模板字段映射错误

**现象**：打车事件悬浮窗显示字面文字"目的地"，而非实际目的地或车牌号。

**根因**：`RuleActionDefaults.kt` 中打车规则的 pending 模板设为 `"目的地"`，但打车规则的 `aiPrompt` 字段定义为 `颜色|车型|车牌`，其中不包含"目的地"。`RuleDisplayTemplateResolver.convertFriendlyPlaceholders()` 无法将"目的地"转为任何 `{fieldN}` 占位符，因此渲染结果为字面文字。

**修复**：将打车规则 pending 模板从 `"目的地"` 改为 `"车牌"`（对应 `{field3}`），渲染后显示实际车牌号如"鄂J·DB7979"。

**排查结果**：其余 7 条内置规则（列车、航班、取件、取餐、取票、寄件、普通日程）模板均正确，无需修改。

**文件**：`core/rule/RuleActionDefaults.kt`

### Bug 2：内置规则模板修改后不生效

**现象**：用户在规则编辑器中修改模板（如将列车待办模板从"检票口 检票"改为"检票口"），保存后仍显示默认模板。

**根因**：`RuleActionSeeder.ensureRuleDefaults()` 对内置规则执行 `stateDao.insertAll(desiredStates)`，无条件覆盖 DB 中的全部状态字段（包括 `displayTemplate`）。保存流程如下：

```
1. ensureRuleDefaults()  → 强制覆盖为默认模板
2. updateDisplayTemplates()  → 写入用户自定义模板
3. RuleRegistry.refresh()
     → ensureDefaults()  → 再次强制覆盖！用户模板丢失
     → RuleDisplayTemplateResolver.refresh()  → 从 DB 读取到的是默认模板
```

步骤 3 中 `ensureDefaults()` 将步骤 2 刚写入的用户模板覆盖回默认值。

**修复**：`ensureRuleDefaults()` 对内置规则改为先读取 DB 中已有状态记录，保留用户自定义的 `displayTemplate`，仅同步结构性字段（`name`、`isTerminal`）。

```kotlin
// 修复前：无条件覆盖
stateDao.insertAll(desiredStates)

// 修复后：保留用户 displayTemplate
val existingStates = stateDao.getByRuleId(rule.ruleId).associateBy { it.stateId }
val merged = desiredStates.map { desired ->
    val existing = existingStates[desired.stateId]
    if (existing != null) {
        desired.copy(displayTemplate = existing.displayTemplate)
    } else {
        desired
    }
}
stateDao.insertAll(merged)
```

**文件**：`core/rule/RuleActionSeeder.kt`

### Bug 3：已过期事件状态标签显示错误

**现象**：列车事件已过期（已发车），悬浮窗中标签显示"已检票"（绿色），而非"已结束"（灰色）。其他已过期的已完成事件（取件、打车等）也存在同类问题，显示"已取件"、"已用车"而非"已结束"。

**根因**：`EventPresenter.resolveStatusLabel()` 的 `when` 分支优先级错误——`isCheckedIn` 和 `isCompleted` 排在 `isExpired` 之前：

```kotlin
// 修复前
event.isCheckedIn -> "已检票"       // ← 即使已过期也命中此分支
event.isCompleted -> terminal.name   // ← 同上
isExpired -> "已结束"                // ← 永远走不到
```

**修复**：将 `isExpired` 判断移到 `isCheckedIn`/`isCompleted` 之前。同步修复 `resolveStatusColor()`，过期事件统一为 MUTED（灰色）。

```kotlin
// 修复后
isExpired -> "已结束"                // ← 过期优先
event.isCheckedIn -> "已检票"
event.isCompleted -> terminal.name
```

**影响范围**：全部 8 种规则类型的状态标签和颜色均受益于此修复。

**文件**：`core/rule/EventPresenter.kt`

## 四、提示词版本号修复

### 问题

数据备份页点击"检查更新"始终提示"检查失败，请稍后重试"。

### 排查过程

1. UI 层 `BackupSettingsPage` → `MainViewModel.checkPromptUpdatesManually()` → `PromptUpdater.check()`
2. `PromptUpdater.check()` 内部执行 HTTP GET 获取云端 JSON，再用 `kotlinx.serialization` 反序列化为 `RemotePrompts`
3. `catch (e: Exception)` 捕获所有异常后返回 `PromptCheckResult.Error("检查失败，请稍后重试")` — 用户看到的错误信息

### 根因

云端 JSON 返回 `"version": 5.1`（Double 类型），但 `RemotePrompts.version` 字段类型为 `Int`。`kotlinx.serialization` 严格类型匹配，无法将 JSON Double 反序列化为 Kotlin Int，抛出 `SerializationException`。

### 修复方案

**方案选择**：最终决定在服务端将版本号改为整数 `6`，客户端同步提升本地版本号至 `6`，无需修改代码逻辑。

**考虑过的方案（未采用）**：

| 方案 | 描述 | 未采用原因 |
|------|------|-----------|
| 自定义 Serializer | 为 `version` 字段添加 `FlexibleIntSerializer`，兼容 Int/Double | 增加代码复杂度 |
| 版本号改为日期格式 | `version` 字段从 Int 改为 String（如 `20260301`） | 改动范围大，所有版本比较逻辑都要改 |
| 服务端改整数 | 将 `5.1` 改为 `6` | **采用** — 最简单，客户端零修改 |

### 客户端改动

| 文件 | 修改内容 |
|------|---------|
| `core/ai/AiPrompts.kt` | `MIN_PROMPT_VERSION` 5 → 6，`defaultPrompts.version` 5 → 6 |

### 云端提示词同步

云端从 v4 同步到 v6，主要更新内容：

- **prompt_header**：新增 ✈️ 航班、🎫 取票、🚚 寄件的 title 和 description 格式规则
- **tag 列表**：从 `general | train | taxi | pickup` 扩展到 `general | train | taxi | flight | pickup | food | ticket | sender`
- **schedule_prompt**：tag 增加 `flight`，description 格式改为 `【日程】...`
- **pickup_prompt**：tag 增加 `food | ticket | sender`，覆盖取餐/取票/寄件场景

### 经验总结

1. **kotlinx.serialization 不做隐式类型转换**：`Json { isLenient = true }` 只允许非标准 JSON 格式（如无引号字符串），不会将 Double 转为 Int
2. **API 契约要和客户端类型严格一致**：服务端返回的 JSON 类型必须与客户端模型字段类型匹配
3. **版本号用整数更稳妥**：避免浮点数带来的精度和类型问题

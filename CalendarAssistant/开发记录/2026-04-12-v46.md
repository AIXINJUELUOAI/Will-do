## 三、短信取件码自动提取修复

### 背景

用户反馈 MIUI 上无法自动提取短信中的快递取件码。排查发现三个问题：

### 问题 1：`BROADCAST_SMS` 权限导致 MIUI 拦截广播

**根因**：AndroidManifest.xml 中 SmsReceiver 声明了 `android:permission="android.permission.BROADCAST_SMS"`，这是签名级权限。MIUI 的电话框架发送 `SMS_RECEIVED` 广播时未携带该权限，广播被系统过滤，SmsReceiver 完全收不到广播。

**修复**：移除该权限声明。`SMS_RECEIVED` 只需要 `RECEIVE_SMS` 运行时权限。

### 问题 2：冷启动时 settings 读取为默认值

**根因**：SmsReceiver 通过 `repository.settings.value`（StateFlow）读取设置。App 冷启动时 StateFlow 尚未从磁盘加载完成，`isSmsMonitoringEnabled` 读到默认值 `false`，直接 return。

**修复**：改为 `SettingsDataSource(context).loadSettings()` 直接从 SharedPreferences 同步读取，不受异步加载影响。

### 问题 3：MIUI 拦截 `SMS_RECEIVED` 广播

即使修复了权限问题，MIUI 仍可能不投递 `SMS_RECEIVED` 广播。需要备用通道。

**方案**：新增 `SmsContentObserver` 监听 `content://sms` 数据库变化，作为 MIUI 兜底通道。

### 实现细节

#### SmsReceiver（广播通道，非 MIUI ROM 兜底）

- 同步从 SharedPreferences 读取设置，解决冷启动问题
- 保留 106 号段过滤
- 添加探针日志便于调试

#### SmsContentObserver（数据库通道，MIUI 兜底）

- 新建 `core/sms/SmsContentObserver.kt`
- 在 `App.onCreate()` 中注册，监听 `content://sms`
- 按 `_id > lastProcessedId` 查询新短信，避免处理历史数据
- 500ms 时间戳拦截合并多次 onChange 通知
- 106 号段过滤
- 通过 SmsAnalysis.parse() 解析取件码，复用现有入库管线
- 与 SmsReceiver 共享去重逻辑（tag + description + 日期）

#### 双通道架构

```
通道 1: SMS_RECEIVED 广播 → SmsReceiver（MIUI 上被拦截，其他 ROM 正常）
通道 2: content://sms → SmsContentObserver（MIUI 兜底通道）
```

### 备用方案：NotificationListenerService

- 已实现 `service/receiver/SmsNotificationListenerService.kt`
- 监听系统短信 App 的通知，提取通知文本解析取件码
- Manifest 未注册（当前不需要，作为未来备用）
- 若 ContentObserver 在某些 ROM 上也不行可随时启用

## 四、日历反向同步循环修复

### 背景

新增 SmsContentObserver 后，每次 App 启动都会触发事件重复。排查发现根因不是 SMS 代码，而是 `CalendarContentObserver` 存在同步循环 bug。

### 问题分析

**循环机制**：

```
CalendarContentObserver 检测到日历变化
  → 触发 CalendarReverseSyncWorker（5s 延迟）
    → CalendarSyncManagerV2 执行反向同步
      → 同步过程修改了系统日历（createEvent/updateEvent）
        → CalendarContentObserver 再次检测到变化
          → 无限循环 → 事件重复
```

**触发条件**：SmsContentObserver 在 `App.onCreate()` 中的注册改变了启动时序，暴露了这个预存的 bug。

### 修复措施

#### 1. CalendarContentObserver 加防抖（根因修复）

- `onChange()` 中用 `Handler.postDelayed(3000ms)` 替代立即回调
- 快速连续的 Provider 通知（一次逻辑变更通常触发 2-4 次）合并为一次
- `unregister()` 时清理 pending callbacks

#### 2. CalendarReverseSyncWorker 延迟增加

- 初始延迟从 5s → 8s，确保在防抖窗口之后才执行
- 与 ContentObserver 的 3s 防抖窗口配合，总延迟约 8s

#### 3. CalendarSyncManagerV2 加并发守卫

- `syncFromCalendar()` 加 `AtomicBoolean` 防重入
- 防止 ContentObserver 触发的同步和定时同步同时执行
- 与 V1 的 `CalendarSyncManager` 已有守卫对齐

## 六、跨日历去重（已有能力确认）

### 背景

多日历同步场景下，同一事件可能存在于小米日历、Google 日历等多个日历中。需要确保 App 不会重复导入。

### 现有机制

`EventDeduplicator.generateFingerprint()` 按 **标题(小写) + 日期 + 时间 + 地点** 生成内容指纹。`syncFromSingleCalendar()` 在导入前检查：

1. 本地活跃事件指纹匹配 → 跳过（跨日历去重）
2. 本地归档事件指纹匹配 → 跳过

此机制已覆盖所有日历来源，无需额外修改。

## 二、悬浮窗便签卡片外形统一

### 背景

- 悬浮窗中日程卡片与便签卡片混排（通过模式切换），但两者收起状态的外形不一致
- 日程卡片有左侧彩色竖条、不同的内边距结构，便签卡片没有竖条且 padding 节奏不同
- 导致同一列表中卡片高度参差不齐，视觉上缺乏秩序感

### 问题分析

对比两种卡片收起状态的布局差异：

| 差异点 | 日程卡片 | 便签卡片（修改前） |
|--------|---------|-------------------|
| 容器 padding | `horizontal = 12.dp`（Row 包裹） | `horizontal = 16.dp, vertical = 12.dp`（Column 包裹） |
| 标题行间距 | `padding(top = 10.dp, bottom = 4.dp)` | 无显式 padding |
| 日期行间距 | `padding(bottom = 10.dp)` | `padding(top = 6.dp)` |
| 标题字体 | `bodyMedium` + Medium/Bold | `bodyLarge` + Bold |
| 彩色竖条 | 有（4dp 宽） | 无 |

### 设计决策

- **不加彩色竖条**：便签和日程是不同类型，通过有无竖条区分身份比统一所有视觉更合理
- **标题字体统一为 `bodyMedium` + `Bold`**：去掉 `bodyLarge`，与日程卡片完全一致，Bold 字重足以区分便签身份
- **间距节奏统一**：采用日程卡片的 `top = 10, bottom = 4` + `bottom = 10` 结构

### 本轮修改

#### 1. 收起状态外形对齐

- 外层 Column 去掉 `vertical = 12.dp`，仅保留 `horizontal = 16.dp`
- 标题行加上 `padding(top = 10.dp, bottom = 4.dp)`，与日程卡片一致
- 日期行从 `padding(top = 6.dp)` 改为 `padding(bottom = 10.dp)`，与日程卡片一致
- 标题字体从 `bodyLarge` 改为 `bodyMedium`，保留 `FontWeight.Bold`

#### 2. 展开状态修复

去掉外层 `vertical` padding 后暴露了两个问题：

1. **待办项贴近底边**：展开区域缺少底部 padding
2. **分界线离日期过远**：日期行 `bottom = 10.dp` 与展开 Column 的 `top = 10.dp` 叠加成 20dp 间距

修复：
- 展开区域 Column 从 `padding(top = 10.dp)` 改为 `padding(bottom = 12.dp)`
- 去掉 `top` padding，由日期行的 `bottom = 10.dp` 提供与日程卡片一致的间距
- 补回 `bottom = 12.dp`，防止内容贴边

### 修改后效果

- **宽度**：两者均为 `fillMaxWidth()`
- **高度**：`10 + bodyMedium + 4 + bodySmall + 10` — 像素级一致
- **圆角**：均为 `RoundedCornerShape(14.dp)`
- **阴影**：均为 `2.dp`
- **区分方式**：日程卡片有左侧彩色竖条，便签没有；便签标题 Bold 区分身份

## 五、悬浮窗识屏按钮改为截屏识屏

### 背景

- 悬浮窗底部输入栏的识屏按钮原功能为启动系统图片选择器上传图片识屏
- 磁贴 `CaptureTileService` 已实现截屏识屏流程（调用无障碍服务 `takeScreenshot`）
- 用户希望悬浮窗识屏按钮也能直接截屏识屏，与磁贴行为一致

### 设计决策

- **识屏后不恢复悬浮窗**：用户明确要求截屏后悬浮窗保持隐藏
- **无障碍服务不可用时回退**：如果无障碍服务未开启，回退到原图片选择器流程，保证功能不降级

### 修改内容

`service/floating/FloatingScheduleService.kt`：
- 将 `startImagePickFlow()` 替换为 `startScreenshotAnalysisFlow()`
- 新流程：隐藏悬浮窗 → 检查无障碍服务 → 可用时调用 `TextAccessibilityService.startAnalysis()` 截屏识屏 → 结果通过通知展示
- 不可用时回退到 `ImagePickHandleActivity` 图片选择

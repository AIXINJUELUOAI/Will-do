# 第一部分：项目概述

## 1.1 项目简介

Will do（智能日历助手）是一款基于 Android 平台的智能日程管理应用，采用 Jetpack Compose 构建现代化 UI，基于 MVVM + Repository 架构实现清晰的数据分层和模块化设计。项目通过 AI 识别、系统日历同步、实况胶囊/超级岛、悬浮窗等能力形成高效的日程管理闭环。

该应用的核心理念是「智能、简洁、实用」：

- 智能：截图/文本/图片识别、AI 解析、结构化日程、规则引擎自动分类
- 简洁：统一入口、减少打扰、自动化流程
- 实用：双向日历同步、实时提醒、跨应用悬浮查看

项目名称 "Will do" 体现主动安排与执行的生活态度，应用通过简洁直观的交互与智能化功能帮助用户更好地管理时间。

### 技术选型说明

**Jetpack Compose** 是 Google 推荐的现代 Android UI 工具包，声明式编程提升界面表达力与动画能力。本项目 100% 使用 Compose 实现 UI。Compose BOM 统一管理版本，避免 UI 组件版本不一致。

**Ktor Client** 提供轻量协程化网络能力，配合 Kotlinx Serialization 自动处理 JSON 编解码，适合 AI 调用场景。通过 ContentNegotiation 插件统一处理 JSON 序列化。

**ML Kit Text Recognition** 在本地完成 OCR 识别，中文识别能力强且不依赖网络，既保证隐私也提升响应速度。

**Kotlinx Serialization** 作为统一序列化层，避免手写解析逻辑，提高数据模型的可维护性。

**Kotlinx Coroutines** 为所有 I/O 与网络任务提供并发支持，保证主线程稳定与可预测的 UI 响应。

**Shizuku / Root** 用于无障碍服务恢复，避免系统重置后无法自动识别的问题。

**Room** 作为本地主存储，取代 JSON 文件存储。通过 Master/Instance 模型对齐系统日历重复事件，配合 TypeConverters 处理复杂类型。启动时自动从旧 JSON 迁移，迁移完成后清理旧文件。

**WorkManager** 用于日历反向同步调度，配合 ContentObserver 防抖触发。

**hyperisland_kit + Xposed** 用于 HyperOS 超级岛展示，模块在 SystemUI 进程内发送岛通知。

### 架构模式：MVVM + Repository

项目采用 MVVM（Model-View-ViewModel）架构并引入 Repository 模式：

- **关注点分离**：UI 层不直接处理业务逻辑。
- **响应式数据流**：StateFlow 驱动 UI 自动更新。
- **统一数据访问**：AppRepository 管理数据读写与缓存。

## 1.4 核心功能一览

### 智能日程识别

支持截图识别、文字输入识别、图片导入识别。文本识别采用双 Prompt 并发模式，多模态识别采用 unified prompt，支持 OpenAI 兼容与 Gemini 视觉接口。AI 结果通过强特征去重（取件码/车牌/车次）和弱特征补充（标题归一化 + 时间容差）消除重复。提示词标签统一使用中文格式（【取件】/【列车】/【用车】等）。

### 规则引擎

内置 8 条规则（日程/列车/打车/航班/取件/取餐/取票/寄件），每条规则定义状态机（状态 + 转换 + 动作）、AI 提示词补丁、展示模板与图标。支持用户自定义规则扩展。EventPresenter 统一渲染入口，消除 `if(tag==xxx)` 硬编码。

### 课程表管理系统

支持课程管理、单双周、排除日期、临时调课、醒课表导入与虚拟日程展示。课程可同步至系统日历（未来 16 周）。

### 实况胶囊与超级岛

支持 Android 原生胶囊（Live Activity）、Flyme 胶囊与 HyperOS 超级岛。胶囊类型覆盖日程、取件、网速、OCR 进度/结果，并支持聚合取件模式。原生胶囊通过 `requestPromotedOngoing` 在 Android 14+ 实现实况通知上岛。

### 悬浮窗与侧边栏

通过音量键长按（动作可配置）或磁贴呼出悬浮窗，支持 AI 文本输入与图片导入。EdgeBar 支持侧边栏手势唤起，并与悬浮窗互斥显示。

### 日历同步

CalendarSync V2 基于 Room Master/Instance 模型，支持 App 与系统日历双向同步（单次 + 重复日程），包含去重、映射维护与低内存保护。

### 稳定性与权限恢复

内置异常日志与轻量 ANR 监测；通过 Shizuku/Root 自动恢复无障碍服务；KeepAlive 定期检查后台状态。

## 2.3 胶囊状态管理与通知架构

CapsuleStateManager 采用「主动唤醒 + 直接通知」模式，独立计算胶囊状态并直接通过 NotificationManager 发布通知（v36 移除了 CapsuleService 前台服务）。

```kotlin
private fun createCapsuleStateFlow(): StateFlow<CapsuleUiState> {
    val tickerTrigger = MutableStateFlow(System.currentTimeMillis())

    appScope.launch {
        while (true) {
            delay(10_000)
            tickerTrigger.value = System.currentTimeMillis()
        }
    }

    val baseCombine = combine(
        repository.events,
        repository.courses,
        repository.settings,
        tickerTrigger,
        forceRefreshTrigger
    ) { events, courses, settings, _, _ ->
        Triple(events, courses, settings)
    }

    return combine(baseCombine, networkSpeedState, ocrCapsuleState) { (events, courses, settings), networkSpeed, ocrCapsule ->
        computeCapsuleState(events, courses, settings, networkSpeed, ocrCapsule)
    }.stateIn(...)
}
```

核心特性：

1. **OCR 胶囊与日程胶囊共存**：OCR 分支通过 `computeScheduleCapsules()` 提取日程胶囊后合并，不再覆盖日程
2. **网速胶囊覆盖其他胶囊（实验室）**
3. **forceRefresh 立即触发重新计算**
4. **直接通知发布**：CapsuleStateManager 内置 NotificationManager，状态变化时直接发布/更新/取消通知

### CapsuleService 移除（v36）

原 CapsuleService 的前台服务模式导致胶囊通知无法被 Android 提升为 Live Activity。v36 将其职责合并到 CapsuleStateManager：

- `startServiceWakeup()` → `startNotificationManager()`：监听 uiState 变化，直接发布通知
- `upsertCapsule()` → `updateCapsules()`：`notificationManager.notify()` + stale 清理
- `cleanupStaleCapsules()` → `cleanupStaleNotifications()`：每 3 秒检查过期通知
- `activeNotifIds` 改为 `ConcurrentHashMap.newKeySet()` 保证线程安全

移除前台服务后，App 依赖用户开启自启动保持后台运行（国产 ROM 基本默认开启给常用 APP）。ongoing 通知在进程被杀后仍保留在通知栏。

### 聚合取件胶囊策略

当开启聚合且同一时间存在多个取件码时，仅显示一个聚合胶囊，并支持批量完成：

```kotlin
val aggregateMode = settings.isPickupAggregationEnabled && pickupEvents.size > 1
```

## 2.5 HyperOS 超级岛 / Xposed 架构

HyperOS 超级岛通过 Xposed 在 SystemUI 进程中发送通知，App 仅负责构建请求。

```
App Process
  CapsuleStateManager → MiuiIslandManager.update()
      → MiuiIslandDispatcher.sendBroadcast(ACTION_SHOW_MIUI_ISLAND)

SystemUI Process (Xposed)
  MiuiIslandDispatcherHook → MiuiIslandDispatcher.register()
      → receive broadcast → HyperIslandNotification → NotificationManager.notify()
```

关键点：

- `SelfHook` 将 `XposedModuleStatus.isActive()` Hook 为 true。
- `MiuiIslandDispatcherHook` 只在 SystemUI 进程注册。
- Island 通道为 `calendar_assistant_island_silent_v2`（静默）。

## 2.6 权限恢复与后台保活架构

```
KeepAliveReceiver (30min)
    ↓
AccessibilityGuardian.restoreIfNeeded()
    ↓
PrivilegeManager (Shizuku / Root)
    ↓
settings put secure enabled_accessibility_services
```

此机制用于系统重置无障碍服务时自动恢复，避免识别功能失效。

# 第四部分：核心功能详解

## 4.1 AI 智能识别

AI 智能识别支持截图识别、文本输入识别、图片导入识别三种方式。截图识别通过无障碍服务截屏，OCR 提取文本后由 AI 解析；多模态模式则直接图片直传。

### 4.1.1 识别流程架构

### 4.1.2 OCR 文本重构与日期锚点

OCR 结果先经过 LayoutAnalyzer 重构，再注入日期锚点，避免跨段落时间混淆：

```kotlin
val reconstructedText = LayoutAnalyzer.reconstructLayout(filteredElements, screenWidth)
val anchoredText = injectDateAnchors(reconstructedText, LocalDate.now())
```

锚点格式示例：

```
[@date=2026-03-22]
```

锚点基于 `[C]` 行解析系统日期，并自动处理“今天/昨天/周几”等相对时间。

### 无障碍截图与通知面板控制

`TextAccessibilityService` 承担截图与面板控制：

- 长按音量+（500ms）触发悬浮窗
- `takeScreenshot` 在主线程执行，分析在后台执行
- 三段式收起面板策略：API 指令 → 广播 → Back 补刀

### 4.1.3 双 Prompt 并发模式

文本 AI 模式下，识别拆分为两条并发 Prompt，Schedule Prompt 动态拼接已启用规则的 aiPrompt 作为补丁：

**Schedule Prompt（普通日程 + 规则补丁）**

```
固定 Header + 固定 Schedule Prompt + 动态规则补丁（isEnabled && appliesToSchedule 的规则）
```

Prompt Header 包含全部 8 个场景的 Title 模板（含 emoji）与 description 格式。

**Pickup Prompt（取件类）**

固定 Prompt，覆盖取件/取餐/取票/寄件四类场景。

所有提示词中标签格式统一为中文：`【取件】`/`【列车】`/`【用车】`/`【航班】`/`【取餐】`/`【取票】`/`【寄件】`/`【日程】`。

### 4.1.4 结果合并与去重（v31 升级）

去重逻辑升级为强特征 + 弱特征两级合并：

**强特征去重**（按微格式提取关键标识）：
- 取件/取餐：提取 `【取件】/【取餐】` 微格式中的取件码
- 用车：提取 `【用车】` 微格式或标题中的车牌
- 列车：提取 `【列车】` 微格式或标题中的车次

**弱特征补充**：
- 标题归一化一致且时间差 ≤ 10 分钟时合并

**保留策略优先级**：
1. 含微格式的事件优先保留
2. 含强特征的事件优先保留
3. 时间解析成功且信息更完整者优先

```kotlin
val finalEvents = deduplicateAiEvents(mergedEvents)
```

### 4.1.5 微格式规范

Description 标签统一使用中文（v34 中文化），AI 提示词输出格式同步更新：

| 类型 | 格式 | 示例 |
|------|------|------|
| 火车票 | `【列车】车次\|检票口\|座位号` | `【列车】G1024\|8A\|15A` |

### 4.1.6 RecognitionProcessor 实现

```kotlin
suspend fun analyzeImage(bitmap: Bitmap, settings: MySettings, context: Context): AnalysisResult<List<CalendarEventData>> {
    if (settings.useMultimodalAi) {
        return analyzeImageWithMultimodal(bitmap, settings, context)
    }

    val visionText = processImageWithMlKit(bitmap)
    val reconstructedText = LayoutAnalyzer.reconstructLayout(...)
    val anchoredText = injectDateAnchors(reconstructedText, LocalDate.now())

    val scheduleDeferred = async { analyzeSchedule(anchoredText, settings, context) }
    val pickupDeferred = async { analyzePickup(anchoredText, settings, context) }

    val scheduleEvents = (scheduleDeferred.await() as? AnalysisResult.Success)?.data ?: emptyList()
    val pickupEvents = (pickupDeferred.await() as? AnalysisResult.Success)?.data ?: emptyList()

    val refinedSchedule = filterRedundantSchedules(scheduleEvents, pickupEvents)
    val merged = mergeAndDeduplicate(refinedSchedule + pickupEvents)
    val finalEvents = deduplicateAiEvents(merged)  // v31 强特征去重
    return AnalysisResult.Success(finalEvents)
}
```

识别结果写入前通过 `normalizeDescriptionHeader()` 将英文 ruleId 标签统一替换为中文名（general→日程、train→列车、taxi→用车、flight→航班、pickup→取件、food→取餐、ticket→取票、sender→寄件）。

### 4.3.2 胶囊状态计算

```kotlin
private fun computeCapsuleState(...): CapsuleUiState {
    if (ocrCapsule != null && settings.isLiveCapsuleEnabled) {
        return CapsuleUiState.Active(listOf(ocrCapsule))
    }

    if (settings.isNetworkSpeedCapsuleEnabled && networkSpeed != null) {
        return CapsuleUiState.Active(listOf(networkSpeedCapsule))
    }

    if (!settings.isLiveCapsuleEnabled) return CapsuleUiState.None

    val activeEvents = allEvents.filter { !it.isCompleted && now.isBefore(endDateTime) }
    val (pickupEvents, scheduleEvents) = activeEvents.partition { it.tag == EventTags.PICKUP }

    if (aggregateMode) capsules.add(aggregatePickupCapsule)
    return if (capsules.isNotEmpty()) CapsuleUiState.Active(capsules) else CapsuleUiState.None
}
```

| 原生胶囊 | App 进程 | Notification (Live) | Android 14+ | 否 | 多条并存 |
| Flyme 胶囊 | App 进程 | Notification + RemoteViews | Flyme + 许可权限 | 否 | 多条并存 |
| HyperOS 超级岛 | SystemUI 进程 | HyperIslandNotification | HyperOS + Xposed 激活 | 否 | 单条优先 |

关键差异：

- **数据路径**：原生/Flyme 直接由 App 构建 Notification；超级岛由 App 构建请求，通过广播交给 SystemUI 再生成通知。
- **展示策略**：原生/Flyme 可以并行展示多个胶囊；超级岛只选择一个目标胶囊（优先级/时间最近）。
- **前台服务**：v36 移除 CapsuleService 后，三种路径均不依赖前台服务。原生胶囊通过 `requestPromotedOngoing` Bundle 请求 Live Activity 提升。

#### 4.3.4.1 原生胶囊路径（NativeCapsuleProvider）

- 使用 `Notification.Builder` + `App.CHANNEL_ID_LIVE` 构建
- API 35+ 通过 `Bundle("android.requestPromotedOngoing", true)` 请求实况通知提升
- API 36+ 通过 `setShortCriticalText()` 在状态栏胶囊中显示简短文本
- 点击打开 `MainActivity`，取件类会附加 `openPickupList=true`
- 动作按钮通过 `EventActionReceiver` 处理（已完成/已检票/已取）
- v36 精简：移除 `setCategory`、`setGroup`、`setForegroundServiceBehavior` 等干扰实况通知提升的配置

#### 4.3.4.2 Flyme 胶囊路径（FlymeCapsuleProvider）

- 仍使用 Notification，但通过 `RemoteViews` 自定义布局
- 布局：`notification_live_flyme.xml` 与 `notification_live_network_speed.xml`
- 关键 extras：`notification.live.capsuleStatus`, `notification.live.capsuleType`, `notification.live.capsuleContent`, `notification.live.capsuleBgColor`, `notification.live.capsuleIcon`, `is_live`, `notification.live.operation`, `notification.live.type`
- 图标使用白色胶囊图标（Icon 白色化）以适配 Flyme 背景
- 依赖权限：`flyme.permission.READ_NOTIFICATION_LIVE_STATE`
- 依赖 Manifest 元数据：`com.meizu.flyme.live_notification=true`

#### 4.3.4.3 HyperOS 超级岛路径（MiuiIslandManager）

- `CapsuleStateManager` 进入岛模式条件：`settings.isLiveCapsuleEnabled && OsUtils.isHyperOS() && XposedModuleStatus.isActive()`
- 进入岛模式时执行：`MiuiIslandManager.update()` 发送岛请求
- 通过广播 `ACTION_SHOW_MIUI_ISLAND` 向 SystemUI 发送请求
- 广播受签名权限保护：`com.antgskds.calendarassistant.SEND_ISLAND`

### 4.3.5 CapsuleStateManager 通知管理（原 CapsuleService 已移除）

v36 将 CapsuleService 的通知管理逻辑合并到 CapsuleStateManager，移除前台服务依赖：

- Provider 选择：`FlymeUtils.isFlyme()` → FlymeCapsuleProvider，否则 NativeCapsuleProvider
- 每条胶囊通过 `notificationManager.notify()` 直接发布
- `activeNotifIds` 使用 `ConcurrentHashMap.newKeySet()` 保证线程安全
- 每 3 秒清理过期胶囊通知，避免残留
- `cleanupStaleNotifications` 清理系统通知与内存映射
- OCR 进度和结果共用通知 ID（88886），分析完成时直接更新而非取消再创建
- OCR 胶囊与日程胶囊共存：OCR 分支通过 `computeScheduleCapsules()` 提取日程后合并

## 4.5 日历同步

### 4.5.1 同步策略（CalendarSync V2）

- 基于 Room Master/Instance 模型的双向同步
- App → System：单次事件 + 重复事件（RRULE/EXDATE）写入，sync_map 维护
- System → App：重复实例缺失 → 写入 `event_excluded_dates` + `isCancelled`
- 同步窗口：前后 30 天
- 过滤规则：仅同步 `eventType == EVENT` 且 `skipCalendarSync == false` 的事件
- 课程同步：单向强制重建，仅同步未来 16 周
- 低内存保护：可用内存低于 64MB 时终止同步并记录日志
- 重复日程同步为默认行为，无需用户手动开启（v34 移除独立开关）

### 4.5.2 同步流程图

```
AppRepository
  ├─ syncAllToCalendar(events, courses)
  │    ├─ CalendarManager.getOrCreateAppCalendar()
  │    ├─ syncCourses (delete+rebuild, 16 weeks ahead)
  │    ├─ syncEvents (CalendarSyncManagerV2: master + instance + sync_map)
  │    └─ 防重映射补建（升级场景：先补齐 calendar_sync_map）
  │
  └─ syncFromCalendar()  (CalendarReverseSyncWorker)
       ├─ queryEventsByIds (update/delete mapped)
       ├─ queryEventsInRange (add new)
       └─ recurringInstances (缺失 → excluded_dates / isCancelled)
```

补充说明：

- **正向同步**：课程采用”删除后重建”的强制策略，普通事件按映射更新或创建。
- **反向同步**：先查询已映射事件更新/删除，再扫描窗口内新增事件并去重。重复实例缺失写入 excluded_dates。
- **防抖**：ContentObserver 触发后通过 WorkManager（OneTimeWork + 5s delay + UniqueWork REPLACE）防抖。
- **定期同步**：`CalendarSyncReceiver` 每 1 分钟兜底触发。
- **升级防重**：V2 正向同步前先补建 `calendar_sync_map`，按 appId/指纹/重复规则匹配系统日历既有事件。

### 4.5.3 重复日程同步

- 重复日程为默认行为（v34 移除 Beta 开关，`allowRecurringSync` 固定为 `true`）
- 仅同步 ±30 天实例，最大 2000 条
- 超过上限会抛出 `RecurringSyncLimitException`

# 第九部分：关键功能实现深析

> 本部分从「怎么做到的」角度深入解析每个功能的技术实现，包含实现原理、关键代码与踩坑记录。

## 9.1 AI 智能识别

### 9.1.1 OCR 文本重构与布局分析

**问题背景**：OCR 输出是一组离散的 Text.Element，丢失了原始布局的行列关系（如聊天气泡左侧 vs 右侧、同一条消息被拆成多段）。

**实现方案**：LayoutAnalyzer 两级重构：

1. **行聚合**：按 Y 坐标容差（屏幕高度 × 阈值）将元素归入 `VisualRow`，每行按 X 排序
2. **块聚合**：相邻行按 X 起始偏移和间距归入 `Block`，识别出逻辑段落
3. **聊天气泡识别**：屏幕比例判定（左侧 < 28%、右侧 > 72%），结合上下文记忆推断发送方

```kotlin
val reconstructedText = LayoutAnalyzer.reconstructLayout(filteredElements, screenWidth)
val anchoredText = injectDateAnchors(reconstructedText, LocalDate.now())
```

日期锚点 `[@date=yyyy-MM-dd]` 注入在系统消息行之后，支持"今天/昨天/周几"等相对时间解析，避免跨段时间混淆。

**踩坑**：早期不做布局重构直接拼接 OCR 文本，导致"10:00 出发"和"14:00 到达"被模型当成同一个事件的起止时间。

### 9.1.2 双 Prompt 并发 + 规则补丁

**问题背景**：单 Prompt 难以同时覆盖日程和取件码两类场景，取件码容易被遗漏或与普通日程混淆。

**实现方案**：拆分为 Schedule Prompt 和 Pickup Prompt 并发执行：

```
Schedule Prompt = 固定 Header + 固定 Schedule Body + 动态规则补丁
Pickup Prompt = 固定取件类 Prompt（稳定优先，不拼规则补丁）
```

Schedule Prompt 的动态规则补丁由 `RulePatchProvider` 生成，仅拼接 `isEnabled && appliesToSchedule` 的规则。Prompt Header 包含全部 8 个场景的 Title 模板（含 emoji）和 description 格式：

```
- 📅 普通日程: title 直接使用
- 🚄 列车: title 格式为 车次号，description 格式为 【列车】车次|检票口|座位号
- 🚗 用车: ...
```

```kotlin
val scheduleDeferred = async { analyzeSchedule(anchoredText, settings, context) }
val pickupDeferred = async { analyzePickup(anchoredText, settings, context) }
```

**踩坑**：早期 Pickup Prompt 也拼规则补丁，导致 AI 偶尔在取件结果中混入列车信息。修复后 Pickup Prompt 固定不变。

### 9.1.3 AI 结果强特征去重

**问题背景**：OCR 识别同一张截图时，AI 可能输出多条语义相同但文字不同的事件（如"顺丰快递 5566"和"5566 菜鸟驿站"）。

**实现方案**（v31 升级）：两级去重

**强特征去重**（按微格式提取关键标识强制合并）：

```kotlin
fun deduplicateAiEvents(events: List<CalendarEventData>): List<CalendarEventData> {
    // 1. 按取件码分组（取件/取餐提取微格式中的号码）
    // 2. 按车牌分组（用车提取微格式中的车牌号）
    // 3. 按车次分组（列车提取微格式中的车次号）
    // 每组保留信息最完整的一条
}
```

**弱特征补充**：标题归一化（去除空格/标点）一致且时间差 ≤ 10 分钟时合并。

**保留策略优先级**：含微格式 > 含强特征 > 信息更完整。

**踩坑**：早期仅按 `title|startTime` 简单分组，无法覆盖"标题不同但语义相同"的场景（如"打车"和"网约车"）。

### 9.1.4 Description 标签中文化

**问题背景**：AI 输出的 description 标签原为英文 ruleId（如 `【pickup】5566|顺丰|驿站`），用户在悬浮窗/通知中看到英文标签不友好。

**实现方案**（v34/v35）：`normalizeDescriptionHeader()` 在写入前统一替换：

```kotlin
fun normalizeDescriptionHeader(description: String, tag: String): String {
    val mapping = mapOf("general" to "日程", "train" to "列车", "taxi" to "用车",
        "flight" to "航班", "pickup" to "取件", "food" to "取餐", "ticket" to "取票", "sender" to "寄件")
    // 替换 【ruleId】 → 【中文名】
}
```

`RuleMatchingEngine.normalizeRuleId()` 已兼容中英文双向映射，确保旧数据仍可正确解析。

### 9.1.5 API 调用与降级重试

**问题背景**：部分模型不支持 `reasoning_effort` 参数，直接返回 400/422 错误。

**实现方案**：`ApiModelProvider` 使用 `ConcurrentHashMap` 缓存每个模型是否支持 reasoning：

```kotlin
private val reasoningSupported = ConcurrentHashMap<String, Boolean>()

// 首次调用带 reasoning_effort
// 400/422 → 标记不支持 → 自动降级重试（不带 reasoning_effort）
```

### 9.1.6 云端 Prompt 更新

**实现方案**：`PromptUpdater` 通过 HTTP GET 拉取远端 Prompt 配置（`RemotePrompts`），比较版本号，支持用户忽略特定版本：

### 9.1.7 多模态图片压缩

## 9.2 规则引擎

### 9.2.1 规则匹配与载荷提取

**问题背景**：事件类型识别早期通过 `if(tag == "train")` 硬编码，新增事件类型需要改多处代码。

**实现方案**：`RuleMatchingEngine` 统一解析链路：

```kotlin
fun resolveRuleId(event: MyEvent): String {
    // 1. 从 description 提取 【ruleId】 前缀
    val (ruleId, payload) = resolvePayload(event.description)
    if (ruleId != null) return normalizeRuleId(ruleId)
    // 2. 回退到 event.tag
    return normalizeRuleId(event.tag)
}

fun normalizeRuleId(raw: String): String {
    // 中文映射："取件"→RULE_PICKUP, "列车"→RULE_TRAIN, ...
    // 英文映射："pickup"→RULE_PICKUP, "train"→RULE_TRAIN, ...
}
```

载荷提取支持 `【ruleId】field1|field2|field3` 格式，`splitFields()` 按 `|` 分割为定长数组。

### 9.2.2 EventPresenter 统一渲染

**问题背景**：悬浮窗、胶囊、通知、主页列表各自实现渲染逻辑（TransportUtils/PickupUtils），维护成本高且容易不一致。

**实现方案**（v34 重构）：`EventPresenter.present()` 作为唯一渲染入口：

```kotlin
fun present(context: Context, event: MyEvent): EventRenderModel {
    val ruleId = RuleMatchingEngine.resolveRuleId(event)
    return when (ruleId) {
        RULE_TRAIN -> resolveTrainDisplay(event, ...)

    }
}
```

- **悬浮窗**：未过期用动态标题，过期回退原始标题

## 9.3 实况胶囊与通知

### 9.3.1 胶囊状态计算

**问题背景**：胶囊需要在日程即将开始、OCR 识别中、网速监控等多种状态下切换，且需要定期刷新（事件过期时自动消失）。

**实现方案**：`CapsuleStateManager` 使用 `combine` 合并多个数据源：

```kotlin
combine(baseCombine, networkSpeedState, ocrCapsuleState) { base, networkSpeed, ocrCapsule ->
    computeCapsuleState(events, courses, settings, networkSpeed, ocrCapsule)
}
```

计算优先级：OCR 胶囊 + 日程胶囊共存 → 网速胶囊覆盖 → 胶囊关闭。

v36 修复了 OCR 胶囊活跃时直接 return 丢弃所有日程胶囊的 BUG，提取 `computeScheduleCapsules()` 方法实现共存：

```kotlin

return CapsuleUiState.Active(listOf(ocrItem) + scheduleCapsules)
```

### 9.3.2 Live Activity 上岛

**问题背景**：CapsuleService 通过 `startForeground()` 将胶囊通知作为前台服务通知发送，Android 不允许前台服务通知被提升为 Live Activity。

**实现方案**（v36）：

1. **移除 CapsuleService**：通知管理合并到 CapsuleStateManager，直接通过 `notificationManager.notify()` 发布
2. **请求提升**：API 35+ 通过 Bundle 请求实况通知提升

```kotlin
// API 35+：请求提升为实况通知
if (Build.VERSION.SDK_INT >= 35) {
    val extras = Bundle().apply {
        putBoolean("android.requestPromotedOngoing", true)
    }
    builder.addExtras(extras)
}

// API 36+：状态栏胶囊显示简短文本
if (Build.VERSION.SDK_INT >= 36) {
    builder.setShortCriticalText(shortText)
}
```

3. **精简干扰项**：移除 `setCategory(CATEGORY_EVENT)`、`setGroup()`、`setForegroundServiceBehavior()` 等阻止提升的配置

**踩坑**：对比参考项目 Hyper-pick-up-code 才发现前台服务是阻止上岛的根因。早期以为是通知渠道或优先级问题，改了多种组合都不生效。

### 9.3.3 CapsuleService 移除

**问题背景**：CapsuleService 作为前台服务运行时，"胶囊服务运行中"通知常驻状态栏，用户反感且阻止 Live Activity 上岛。

**实现方案**（v36）：将 CapsuleService 的全部职责合并到 CapsuleStateManager：

| 原方法 | 新方法 | 说明 |
|--------|--------|------|
| `startServiceWakeup()` | `startNotificationManager()` | 监听 uiState 变化 |
| `upsertCapsule()` | `updateCapsules()` | notify + stale 清理 |
| `cleanupStaleCapsules()` | `cleanupStaleNotifications()` | 每 3 秒检查过期 |

`activeNotifIds` 从 `mutableSetOf` 改为 `ConcurrentHashMap.newKeySet()`，修复多协程并发读写导致的 `ConcurrentModificationException`（归档事件时偶现崩溃）。

移除前台服务后，App 依赖用户开启自启动保持后台运行。ongoing 通知在进程被杀后仍保留在通知栏（只是不会自动更新）。

### 9.3.4 Flyme 胶囊适配

### 9.3.5 HyperOS 超级岛

## 9.4 日历同步

### 9.4.1 CalendarSync V2 双向同步

**问题背景**：旧版同步仅支持单次事件，重复日程无法同步，且反向同步依赖定时轮询。

**实现方案**：基于 Room Master/Instance 模型：

**正向同步（App → System）**：
- Master 有 rrule → 写 Events 表（RRULE/EXDATE）
- 单次事件 → 写单条
- 保存 `calendar_sync_map` 映射

**反向同步（System → App）**：
- 通过 `CalendarContract.Instances` 查询
- 重复实例缺失 → 写入 `event_excluded_dates` + 标记 `isCancelled`
- 防抖：ContentObserver 触发后通过 WorkManager（OneTimeWork + 5s delay + UniqueWork REPLACE）

**踩坑**：ContentObserver 在国产 ROM 上触发频率极高（MIUI 每次 ContentResolver.notifyChange 触发多次），不防抖会导致短时间内大量重复同步。

### 9.4.2 升级防重同步

**问题背景**：用户从旧版升级到 V2 后，已有系统日历事件缺少 `calendar_sync_map` 映射，V2 正向同步会将已有事件再创建一份。

**实现方案**（v33）：V2 正向同步前先执行两层补建：

1. 先把旧 `sync_data.json` 中已有 mapping 补写进 `calendar_sync_map`
2. 再扫描系统日历内已有事件，优先按 `appId` 匹配，兜底按指纹/重复规则基线哈希匹配

首次自动同步时若发现还没有映射，会先做一次反向吸收，再进入正向写入。

## 9.5 Room 主存与迁移

### 9.5.1 Master/Instance 模型

**问题背景**：旧 JSON 存储将重复日程的每个实例都独立存储，与系统日历的 Events/Instances 模型不对齐，导致同步时无法正确处理 RRULE。

**实现方案**：Room 数据库采用 Master/Instance 分离：

- `event_masters`：存储事件基本信息和 RRULE
- `event_instances`：存储每次具体发生的时间、状态、完成/归档时间
- `event_excluded_dates`：存储被排除的实例时间

UI 只展示 instance，不直接展示 master。`isCancelled` 的实例默认不展示。自动归档只归档实例（`endTime < now - 24h`），不影响后续实例。

### 9.5.2 JSON → Room 迁移

**实现方案**（v33）：

1. App 启动时 `refreshData()` 主动迁移活动/重复/归档日程
2. 每条 JSON → master + instance，重复规则拆分为 master/instance/excludedDates
3. 迁移完成后立即删除旧文件：`events.json`、`archives.json` 及旧备份
4. 仅保留 `*.room.bak` 作为应急兜底
5. `master.source = "legacy_json"` 标记迁移来源

**踩坑**：早期依赖用户进入归档页才触发迁移，导致"活动已进库、归档仍留在 JSON"的半完成状态。v33 改为启动即全量迁移。

## 9.7 无障碍与音量键

### 9.7.1 长按音量+拦截

**问题背景**：早期方案在 `onKeyEvent(ACTION_UP)` 时判断按压时长，但此时系统已处理了音量增加，无法阻止。

**实现方案**（v36 重构）：完全拦截 + 短按补偿：

```kotlin
override fun onKeyEvent(event: KeyEvent): Boolean {
    if (event.keyCode == KEYCODE_VOLUME_UP) {
        when (event.action) {
            ACTION_DOWN -> {
                // 启动协程计时器（400ms）
                // 完全拦截，不传递给系统
                return true
            }
            ACTION_UP -> {
                if (!longPressTriggered) {
                    // 短按：由 App 手动增加音量并显示系统音量条
                    audioManager.adjustVolume(STREAM_MUSIC, ADJUST_RAISE)
                }
                return true
            }
        }
    }
}
```

长按动作可配置为：识屏 / 悬浮窗 / 无操作。若选择"悬浮窗"但悬浮窗总开关未启用，则直接无操作处理。

**踩坑**：不能只靠 `onKeyEvent` 的 `event.eventTime - event.downTime` 判断，因为 `ACTION_UP` 时系统已经完成了短按音量增加。必须在 `ACTION_DOWN` 时就拦截并自行管理全部按键逻辑。

### 9.11.3 精确过期定时器

**问题背景**：`MainViewModel` 原先每 10 秒轮询刷新 UI，大部分刷新无实际变化，浪费 CPU。

**实现方案**（v35）：计算最近事件过期时间点，只在事件实际过期时触发：

```kotlin
fun calculateDelayToNextExpiration(events: List<MyEvent>): Long {
    val now = System.currentTimeMillis()
    val nextExpiry = events
        .filter { !it.isCompleted && it.endDateTime > now }
        .minOfOrNull { it.endDateTime - now }
        ?: 60_000L  // 无未过期事件时保底 60 秒
    return nextExpiry.coerceIn(1000L, 60_000L)
}
```

大幅减少无效 UI 重组，滑动更流畅。

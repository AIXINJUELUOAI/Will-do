# 第一部分：项目概述

## 1.1 项目简介

Will do（智能日历助手）是一款基于 Android 平台的智能日程管理应用，采用 Jetpack Compose 构建现代化 UI，基于 MVVM + Repository 架构实现清晰的数据分层和模块化设计。项目通过 AI 识别、系统日历同步、实况胶囊/超级岛、悬浮窗等能力形成高效的日程管理闭环。

该应用的核心理念是「智能、简洁、实用」：

- 智能：截图/文本/图片识别、AI 解析、结构化日程
- 简洁：统一入口、减少打扰、自动化流程
- 实用：双向日历同步、实时提醒、跨应用悬浮查看

项目名称 "Will do" 体现主动安排与执行的生活态度，应用通过简洁直观的交互与智能化功能帮助用户更好地管理时间。

### 技术选型说明

**Jetpack Compose** 是 Google 推荐的现代 Android UI 工具包，声明式编程提升界面表达力与动画能力。本项目 100% 使用 Compose 实现 UI。Compose BOM 统一管理版本，避免 UI 组件版本不一致。

**Ktor Client** 提供轻量协程化网络能力，配合 Kotlinx Serialization 自动处理 JSON 编解码，适合 AI 调用场景。通过 ContentNegotiation 插件统一处理 JSON 序列化。

**ML Kit Text Recognition** 在本地完成 OCR 识别，中文识别能力强且不依赖网络，既保证隐私也提升响应速度。

**Kotlinx Serialization** 作为统一序列化层，避免手写解析逻辑，提高数据模型的可维护性。

**Kotlinx Coroutines** 为所有 I/O 与网络任务提供并发支持，保证主线程稳定与可预测的 UI 响应。

**Shizuku / Root** 用于无障碍服务恢复，避免系统重置后无法自动识别的问题。

**hyperisland_kit + Xposed** 用于 HyperOS 超级岛展示，模块在 SystemUI 进程内发送岛通知。

### 架构模式：MVVM + Repository

项目采用 MVVM（Model-View-ViewModel）架构并引入 Repository 模式：

- **关注点分离**：UI 层不直接处理业务逻辑。
- **响应式数据流**：StateFlow 驱动 UI 自动更新。
- **统一数据访问**：AppRepository 管理数据读写与缓存。

## 1.4 核心功能一览

### 智能日程识别

支持截图识别、文字输入识别、图片导入识别。文本识别采用双 Prompt 并发模式，多模态识别采用 unified prompt，支持 OpenAI 兼容与 Gemini 视觉接口。

### 课程表管理系统

支持课程管理、单双周、排除日期、临时调课、醒课表导入与虚拟日程展示。课程可同步至系统日历（未来 16 周）。

### 实况胶囊与超级岛

支持 Android 原生胶囊、Flyme 胶囊与 HyperOS 超级岛。胶囊类型覆盖日程、取件、网速、OCR 进度/结果，并支持聚合取件模式。

### 悬浮窗与侧边栏

通过音量键长按或磁贴呼出悬浮窗，支持 AI 文本输入与图片导入。EdgeBar 支持侧边栏手势唤起，并与悬浮窗互斥显示。

### 日历同步

支持 App 与系统日历双向同步，包含去重、映射维护、重复日程同步（Beta）与低内存保护。

### 稳定性与权限恢复

内置异常日志与轻量 ANR 监测；通过 Shizuku/Root 自动恢复无障碍服务；KeepAlive 定期检查后台状态。

## 2.3 胶囊状态管理架构

CapsuleStateManager 采用「主动唤醒」模式，独立计算胶囊状态，CapsuleService 仅负责展示。

```kotlin
private fun createCapsuleStateFlow(): StateFlow<CapsuleUiState> {
    val tickerTrigger = MutableStateFlow(System.currentTimeMillis())

    appScope.launch {
        while (true) {
            delay(10_000)
            tickerTrigger.value = System.currentTimeMillis()
        }
    }

    val baseCombine = combine(
        repository.events,
        repository.courses,
        repository.settings,
        tickerTrigger,
        forceRefreshTrigger
    ) { events, courses, settings, _, _ ->
        Triple(events, courses, settings)
    }

    return combine(baseCombine, networkSpeedState, ocrCapsuleState) { (events, courses, settings), networkSpeed, ocrCapsule ->
        computeCapsuleState(events, courses, settings, networkSpeed, ocrCapsule)
    }.stateIn(...)
}
```

核心特性：

1. **OCR 胶囊优先级最高**
2. **网速胶囊覆盖其他胶囊（实验室）**
3. **forceRefresh 立即触发重新计算**
4. **主动唤醒服务**：状态变为 Active 时启动 CapsuleService

### CapsuleService Dumb Service 模式

CapsuleService 只负责展示通知，不进行业务计算：

```kotlin
class CapsuleService : Service() {
    private val repository by lazy { AppRepository.getInstance(applicationContext) }
    private val capsuleManager by lazy { repository.capsuleStateManager }

    override fun onCreate() {

                    is CapsuleUiState.None -> cancelAllNotifications()
                }
            }
        }
    }
}
```

## 2.4 HyperOS 超级岛 / Xposed 架构

HyperOS 超级岛通过 Xposed 在 SystemUI 进程中发送通知，App 仅负责构建请求。

```
App Process
  CapsuleStateManager → MiuiIslandManager.update()
      → MiuiIslandDispatcher.sendBroadcast(ACTION_SHOW_MIUI_ISLAND)

SystemUI Process (Xposed)
  MiuiIslandDispatcherHook → MiuiIslandDispatcher.register()
      → receive broadcast → HyperIslandNotification → NotificationManager.notify()
```

关键点：

- `SelfHook` 将 `XposedModuleStatus.isActive()` Hook 为 true。
- `MiuiIslandDispatcherHook` 只在 SystemUI 进程注册。
- Island 通道为 `calendar_assistant_island_silent_v2`（静默）。

## 2.5 权限恢复与后台保活架构

```
KeepAliveReceiver (30min)
    ↓
AccessibilityGuardian.restoreIfNeeded()
    ↓
PrivilegeManager (Shizuku / Root)
    ↓
settings put secure enabled_accessibility_services
```

此机制用于系统重置无障碍服务时自动恢复，避免识别功能失效。

### 4.1.6 RecognitionProcessor 实现

```kotlin
suspend fun analyzeImage(bitmap: Bitmap, settings: MySettings, context: Context): AnalysisResult<List<CalendarEventData>> {
    if (settings.useMultimodalAi) {
        return analyzeImageWithMultimodal(bitmap, settings, context)
    }

    val visionText = processImageWithMlKit(bitmap)
    val reconstructedText = LayoutAnalyzer.reconstructLayout(...)
    val anchoredText = injectDateAnchors(reconstructedText, LocalDate.now())

    val scheduleDeferred = async { analyzeSchedule(anchoredText, settings, context) }
    val pickupDeferred = async { analyzePickup(anchoredText, settings, context) }

    val scheduleEvents = (scheduleDeferred.await() as? AnalysisResult.Success)?.data ?: emptyList()
    val pickupEvents = (pickupDeferred.await() as? AnalysisResult.Success)?.data ?: emptyList()

    val refinedSchedule = filterRedundantSchedules(scheduleEvents, pickupEvents)
    return AnalysisResult.Success(mergeAndDeduplicate(refinedSchedule + pickupEvents))
}
```

### 4.3.3 胶囊展示适配

- **NativeCapsuleProvider**：Android 14+ 原生 Live Activity
- **FlymeCapsuleProvider**：Flyme 实况通知 (RemoteViews)
- **HyperOS 超级岛**：由 MiuiIslandManager 构建请求，MiuiIslandDispatcher 在 SystemUI 发送

### 4.3.4 三种展示路径差异（Native / Flyme / HyperOS）

| 路径 | 运行进程 | 展示载体 | 依赖条件 | 是否依赖 CapsuleService | 展示数量 |
|------|----------|----------|----------|-------------------------|----------|
| 原生胶囊 | App 进程 | Notification (Live) | Android 14+ | 是 | 多条并存 |
| Flyme 胶囊 | App 进程 | Notification + RemoteViews | Flyme + 许可权限 | 是 | 多条并存 |
| HyperOS 超级岛 | SystemUI 进程 | HyperIslandNotification | HyperOS + Xposed 激活 | 否 | 单条优先 |

关键差异：

- **数据路径**：原生/Flyme 直接由 App 构建 Notification；超级岛由 App 构建请求，通过广播交给 SystemUI 再生成通知。
- **展示策略**：原生/Flyme 可以并行展示多个胶囊；超级岛只选择一个目标胶囊（优先级/时间最近）。
- **服务依赖**：原生/Flyme 依赖 CapsuleService 前台服务；超级岛在启用时会主动停止 CapsuleService，避免重复显示。

#### 4.3.4.1 原生胶囊路径（NativeCapsuleProvider）

- 使用 `Notification.Builder` + `App.CHANNEL_ID_LIVE` 构建
- 统一分组 `LIVE_CAPSULE_GROUP`，支持多胶囊并存
- `setShortCriticalText` / `setRequestPromotedOngoing` 用于触发系统 Live 表现
- `setForegroundServiceBehavior(IMMEDIATE)` 优先级提升
- 点击打开 `MainActivity`，取件类会附加 `openPickupList=true`
- 动作按钮通过 `EventActionReceiver` 处理（已完成/已检票/已取）

#### 4.3.4.2 Flyme 胶囊路径（FlymeCapsuleProvider）

- 仍使用 Notification，但通过 `RemoteViews` 自定义布局
- 布局：`notification_live_flyme.xml` 与 `notification_live_network_speed.xml`
- 关键 extras：`notification.live.capsuleStatus`, `notification.live.capsuleType`, `notification.live.capsuleContent`, `notification.live.capsuleBgColor`, `notification.live.capsuleIcon`, `is_live`, `notification.live.operation`, `notification.live.type`
- 图标使用白色胶囊图标（Icon 白色化）以适配 Flyme 背景
- 依赖权限：`flyme.permission.READ_NOTIFICATION_LIVE_STATE`
- 依赖 Manifest 元数据：`com.meizu.flyme.live_notification=true`

#### 4.3.4.3 HyperOS 超级岛路径（MiuiIslandManager）

- `CapsuleStateManager` 进入岛模式条件：`settings.isLiveCapsuleEnabled && OsUtils.isHyperOS() && XposedModuleStatus.isActive()`
- 进入岛模式时执行：`MiuiIslandManager.update()` 发送岛请求，并主动 `stopService(CapsuleService)` 避免双重展示
- 通过广播 `ACTION_SHOW_MIUI_ISLAND` 向 SystemUI 发送请求
- 广播受签名权限保护：`com.antgskds.calendarassistant.SEND_ISLAND`

### 4.3.5 CapsuleService 前台与多胶囊调度

CapsuleService 负责原生/Flyme 胶囊的生命周期管理：

- 启动时先创建占位前台通知（避免服务被系统杀死）
- Provider 选择：`FlymeUtils.isFlyme()` → FlymeCapsuleProvider，否则 NativeCapsuleProvider
- 每条胶囊保存到 `activeCapsules` 映射，ID 来自 `CapsuleItem.notifId`
- 通过 `LIVE_CAPSULE_GROUP` 分组，支持多胶囊并存
- 前台通知会在多胶囊中自动选择“最新且有效”的胶囊作为前台
- 聚合模式下每 3 秒清理过期胶囊，避免残留通知
- `cleanupInvalidNotifications` / `cleanupStaleCapsules` 清理系统通知与内存映射

前台策略补充：

- 首次启动使用占位 ID（`PLACEHOLDER_FOREGROUND_ID`），随后替换为真实胶囊 ID
- Android 14+ 使用 `FOREGROUND_SERVICE_TYPE_SPECIAL_USE` 启动前台

## 4.5 日历同步

### 4.5.1 同步策略

- 默认双向同步：App → 系统日历；系统日历变化通过 ContentObserver + 定时同步回流
- 同步窗口：前后 30 天
- 过滤规则：仅同步 `eventType == EVENT` 且 `skipCalendarSync == false` 的事件
- 课程同步：单向强制重建，仅同步未来 16 周
- 低内存保护：可用内存低于 64MB 时终止同步并记录日志

### 4.5.2 同步流程图

```
AppRepository
  ├─ syncAllToCalendar(events, courses)
  │    ├─ CalendarManager.getOrCreateAppCalendar()
  │    ├─ syncCourses (delete+rebuild, 16 weeks ahead)
  │    └─ syncEvents (create/update + mapping)
  │
  └─ syncFromCalendar()
       ├─ queryEventsByIds (update/delete mapped)
       ├─ queryEventsInRange (add new)
       └─ recurringInstances (beta, limit 2000)
```

补充说明：

- **正向同步**：课程采用“删除后重建”的强制策略，普通事件按映射更新或创建。
- **反向同步**：先查询已映射事件更新/删除，再扫描窗口内新增事件并去重。
- **内容观察者**：`CalendarContentObserver` 监听系统日历变化并防抖触发同步。
- **定期同步**：`CalendarSyncReceiver` 每 1 分钟兜底触发。

### 4.5.3 重复日程同步（Beta）

- 开关 `isRecurringCalendarSyncEnabled` 控制重复日程回流
- 仅同步 ±30 天实例，最大 2000 条
- 超过上限会抛出 `RecurringSyncLimitException`

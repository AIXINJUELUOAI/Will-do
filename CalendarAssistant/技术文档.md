# 第一部分：项目定位与功能边界

## 1.1 项目定位

Will do 是一款 Android 智能日程助手，核心目标是把日程相关信息从“识别输入”稳定转成“可提醒、可同步、可展示、可回写”的业务事实。

核心闭环：

```text
识别输入 -> 结构化草稿 -> 入库中心 -> Room 主存 -> 系统日历同步
       -> Alarm/Notification -> 实况胶囊/桌面小组件/悬浮窗 -> 用户动作回写
```

主要能力：

- 文本、截图、图片、多模态 AI 识别。
- 远程 AI 与本地 LiteRT 模型双路径设计。
- 短信、剪贴板取件/取餐/取票/寄件码类识别。
- 普通日程、重复日程、课程、便签统一投影展示。
- Room 主存与 Android Calendar Provider 双向同步。
- 普通提醒、天气预警、原生实况通知、Flyme 胶囊、HyperOS 超级岛。
- 悬浮窗、EdgeBar 侧边栏、快捷磁贴、长按音量键入口。
- 天气详情、风险提醒、桌面小组件、附件、备份、诊断日志。
- UI 触感反馈、预测性返回、自定义底部弹窗动效。

## 1.2 当前核心结论

```text
Room events.db 是 App 唯一可靠业务事实源
Calendar Provider 是系统日历互通层
Alarm/Notification 是提醒执行层
Center/API/Node 是副作用与架构边界
UI Motion/Haptics 是纯 UI 层能力，不进入 Center/Node
```

Provider-only 方向已经放弃。系统 Calendar Provider 适合承载时间、RRULE、EXDATE、提醒等日历事实，但不适合作为 App 唯一事实源。`tag`、完成、归档、检票、Undo、附件、便签、课程元数据、Widget 快照和天气风险都必须由 App 自己维护。

- `buildConfig = true`：注入 `CODE_AUTHOR`、`BUILD_TIMESTAMP`、`PROMPT_UPDATE_URL`。

- `minSdk = 33`：当前功能以 Android 13+ 为基线，通知权限、预测性返回和实况通知均围绕较新系统能力设计。
- `targetSdk = 36`：原生实况通知和 promoted ongoing 以 Android 16 相关 API 为目标。
- 本地模型能力分支需要 `implementation(project(":ai-engine"))`，并在 `settings.gradle.kts` 中 `include(":ai-engine")`。

## 5.2 `Event` 主实体

路径：`calendar/models/Event.kt`

`Event` 是 Room 表 `events` 的实体，也是 App 日程业务主存的核心模型。

关键字段：

| 字段 | 作用 |
|------|------|
| `id` | Room 自增主键。 |
| `title/location/description` | 标题、地点、备注；特殊类型用 description 微格式。 |
| `startTS/endTS` | 秒级开始/结束时间。 |
| `reminder1/2/3Minutes` | 最多三个普通提醒。 |
| `rrule` | 重复规则；非空且 `parentId == 0` 表示重复母事件。 |
| `exdates` | 被排除的重复实例。 |
| `parentId` | 单次修改/例外子事件的父事件 ID。 |
| `state` | pending/completed/checked-in 等业务状态。 |
| `tag` | general/train/flight/taxi/pickup/food/ticket/sender/course/note。 |
| `archivedAt` | 归档时间戳，null 表示未归档。 |
| `importId/source` | 系统日历映射和来源标记。 |

## 5.3 `EventAttachment` 附件实体

路径：`calendar/models/EventAttachment.kt`

表：`event_attachments`

| 字段 | 作用 |
|------|------|
| `id` | Room 自增主键。 |
| `eventId` | 已保存日程的 Room ID，可为空。 |
| `eventKey` | 新建未保存日程的兜底绑定键。 |
| `localPath` | App 私有目录中的附件副本路径。 |
| `displayName` | 展示名。 |
| `mimeType` | MIME 类型，用于图片预览判断。 |
| `sizeBytes` | 文件大小。 |
| `source` | `manual` 或 `recognition_image`。 |
| `createdAt` | 创建时间。 |

附件绑定策略：

```text
新建日程未保存
   |
   v
附件先绑定 event_key(title + start + end + timezone)
   |
   v
日程保存后拿到 Room eventId
   |
   v
EventAttachmentManager.rebindAttachmentsToEventId
   |
   v
后续优先 event_id 查询，event_key 兜底并自动修复
```

历史 `[img:...]` 来源图标记已经迁移为附件，`description` 回归纯备注和规则微格式，不再承担本地图片路径存储。

```text
        +--> 普通展示：按 RRULE 展开 occurrence
        |
        +--> 删除/编辑 THIS：父事件追加 EXDATE，并物化一个子事件或删除该 occurrence
        |
        +--> 编辑 FUTURE：旧父事件截断，新父事件承接未来规则
        |
        +--> 编辑 ALL：直接更新父事件
```

约束：

- 单次修改必须使用 `EXDATE + 子事件` 表达，不直接改父事件实例。
- 课程本质是重复日程，使用 `tag = course`。
- `tag = course` 不参与完成、检票等普通状态变更。
- 恢复归档重复实例时，需要移除父事件 EXDATE 或恢复子事件归档状态。

```text

   |       +--> LiteRtAiEngineClient.prepare(model)

   |

   |

   +--> LiteRT Engine 初始化
   +--> 文本/图片推理
   +--> AiEngineResult / AiEngineException
   |

   |
   v

```

本地模型设计要点：

- 本地模型服务运行在 `:ai_engine` 进程，避免主进程被模型内存波动拖垮。
- App 侧通过 `LiteRtAiEngineClient` 使用 `Messenger` 与服务通信。
- `prepare()` 用于提前加载模型，图片识别时与 OCR 并行。
- `generate()` 支持文本与单图多模态。
- `AiEngineService` 使用前台服务通知，支持取消请求。
- 服务端会写 `AI_ENGINE` 日志，便于诊断本地模型加载、推理、崩溃和超时。
- 若 `useMultimodalAi` 开启但模型不支持图片，则直接返回明确失败。
- 若 OCR 为空但模型支持多模态，则使用统一多模态 Prompt + 原图兜底。

# 第十七部分：构建验证与维护规则

## 17.1 常用验证命令

```bash
./gradlew.bat :app:compileDebugKotlin
./gradlew.bat :app:assembleDebug
./gradlew.bat :app:assembleRelease
```

完整本地模型能力分支：

```bash
./gradlew.bat :app:compileDebugKotlin :ai-engine:compileDebugKotlin
```

## 17.2 架构护栏

维护时遵守：

1. UI 不直接写 Room、不直接写 Calendar Provider。
2. Service/Receiver/Worker 不绕过 Center 或 Store 主链执行业务写入。
3. 日程持久化必须走 `ScheduleCenter -> CalendarCenter -> StoreDispatcher -> StoreRootNode`。
4. 重复日程单次变更必须用 `EXDATE + 子事件`。
5. 附件不能写入 `description`，必须走 `EventAttachmentManager`。
6. 特殊类型展示继续复用 `tag + description 微格式`，避免新增不必要字段。
7. 普通通知走 `NotificationCenter`，实况状态走 `CapsuleCenter -> CapsuleStateManager`。
8. 天气刷新走 `WeatherRepository`，UI 不直接请求和风接口。
9. 小组件刷新走 `WidgetCenter`，Renderer 只做 RemoteViews 渲染。
10. 日志导出走 `DiagnosticLogCenter`，不要在 UI 中直接拼文件路径。
11. UI Motion 和 Haptics 只属于 UI 层，不进入 Center/Node。
12. 本地模型能力恢复时，必须同时恢复 `:ai-engine`、`LocalSemanticProvider`、`LiteRtAiEngineClient`、Manifest 服务和 ProGuard/R8 规则。

## 17.3 已知风险

- 历史来源截图如果已经被系统日历同步覆盖，附件迁移无法自动恢复原图。
- 小组件 RemoteViews 在不同 Launcher 的尺寸和圆角裁剪表现仍需真机验证。
- 天气预警和实况通知在 HyperOS、Flyme、原生 Android 上需要设备侧回归。
- 剪贴板后台监听依赖 Shizuku/Root 和 ROM logcat 行为，不保证所有 ROM 稳定。
- 本地模型推理受内存、GPU 后端、模型格式和系统前台服务策略影响，需要单独压力测试。
- 标准版移除本地模型入口后，如果恢复完整能力，需要同步恢复构建模块、设置页和日志入口。

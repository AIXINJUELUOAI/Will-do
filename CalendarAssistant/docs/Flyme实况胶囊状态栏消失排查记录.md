# Flyme 实况胶囊状态栏消失排查记录

## 现象

Flyme 用户反馈：实况胶囊通知创建后，状态栏中的胶囊会在日程结束前自行消失。

反馈描述：

- “实况通知在日程到期前就消失了”
- “最快创建日程 0 分钟就消失了”

目前小米/HyperOS 设备未复现。

## 当前判断

暂不确定是 Flyme 系统行为，还是应用通知链路误取消导致。

代码侧存在可疑点：普通通知、胶囊通知、旧提醒残留清理可能共用或误操作同一通知 ID，导致活跃胶囊被提前取消。

## 高可疑代码路径

- `CapsuleStateManager.kt`
  - 单次日程胶囊通知 ID 使用 `eventId.hashCode()`。
  - `updateCapsules()` 负责 `notify(item.notifId, notification)`。

- `ReminderStoreNode.kt`
  - `rebuildForEvent()` 在胶囊开启时会清理普通提醒残留。
  - `cancelDisplayedNotificationsForEvent()` 可能取消 `eventId.hashCode()`。
  - 该 ID 与胶囊通知 ID 存在重叠风险。

- `ReminderReconcileReceiver.kt`
  - 启动后会立即触发一次提醒对账。
  - 后续每 15 分钟对账一次。
  - 对账链路可能间接触发误取消。

- `EventReminderReceiver.kt`
  - 胶囊开启时会跳过普通提醒。
  - 旧提醒广播路径中存在取消通知的行为，需要避免误伤胶囊。

## 暂不修改原因

本次只记录问题，暂不直接修复，避免引入新的通知回归问题，例如：

- 删除、完成、归档后胶囊残留。
- 普通提醒通知无法清理。
- 重复日程实例提醒异常。
- Flyme 私有实况通知参数兼容性变化。

## 后续修复思路

优先采用“取消范围分级”，不要粗暴删除 `cancel` 逻辑。

建议方案：

1. 区分普通提醒通知 ID 与胶囊通知 ID。
2. 删除、完成、归档、过期时允许取消胶囊。
3. 胶囊开启后跳过普通提醒时，只清理普通提醒，不取消当前 live 胶囊。
4. 提醒对账时只重建闹钟和普通提醒，不误杀胶囊。
5. 增加通知日志，记录 `notify` / `cancel` 的通知 ID、事件 ID、原因。
6. 如修复后 Flyme 仍异常，再单独排查 `FlymeCapsuleProvider` 的私有 extras。

## 需要验证的场景

- Flyme：创建即将开始的日程，观察胶囊是否立刻消失。
- Flyme：等待 15 分钟提醒对账后，观察胶囊是否消失。
- 胶囊开启：删除、完成、归档日程后，胶囊应正常消失。
- 胶囊关闭：普通通知提醒应正常出现。
- 重复日程：实例提醒和删除实例都不应产生残留通知。
